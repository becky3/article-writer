## タスク

技術記事 markdown の品質を、観点別に独立した `general-purpose` サブエージェントで並列レビューする。各エージェントが指定観点のみに絞ることで、観点が交差せず重複のない指摘を得る。

`doc-reviewer` エージェントは仕様書レビュー専用の観点を持っており、記事には合わない。本スキルは記事 markdown 専用の汎用観点レビュー手段として機能する。

## 適用範囲

- **対象**: 記事 markdown（`articles/zenn/*.md` を主想定）
- **媒体別ガイドラインの切替**: ガイドライン準拠チェック観点は、対象ファイルパスに応じて参照する `quality-guidelines.md` を切り替える。現在は `articles/zenn/*`（→ `.claude/skills/write-zenn/quality-guidelines.md`）と `articles/hatena/*`（→ `.claude/skills/write-hatena-diary/quality-guidelines.md`）に対応
- **将来別媒体（Qiita, note 等）が追加された場合**: 本 skill 自体は流用するが、ガイドライン切替テーブル（後述「処理手順 3. ガイドライン解決」）にエントリを追加する必要がある。媒体ごとの専用 skill 化はしない方針
- **対象外**: 仕様書・README・コード等の非記事文書（`doc-reviewer` / `code-reviewer` を使用）

## 引数

`$ARGUMENTS` の形式:

- **`<対象ファイルパス>`**: 必須。レビュー対象の記事 markdown ファイルへのパス。**観点 5（ガイドライン準拠チェック）を含む場合はリポジトリルートからの相対パス必須**（絶対パスは観点 5 のパス解決でエラー停止する。詳細は「処理手順 3. ガイドライン解決」参照）。観点 5 を含まない場合は絶対パスでも動作する
- **`<対象ファイルパス> [観点1,観点2,...]`**: 第 2 引数（観点リスト）は任意。指定時は **カンマ区切り**（スペース不可）で渡す。省略時はデフォルト 5 観点全件で並列起動

例:

- `/multi-perspective-review articles/zenn/20260505-xxx.md`
- `/multi-perspective-review articles/zenn/20260505-xxx.md 構成,読者視点`
- `/multi-perspective-review articles/zenn/20260505-xxx.md 文体`（単一観点はカンマ不要）

第 2 引数全体が 1 トークン。観点識別子に含まれる `・` は区切り文字として扱わない（`構成・主題整合性,読者視点` のように正式名 + カンマ区切りも許容）。

引数なし、または 1 つ目の引数がファイルとして存在しない場合はエラーを表示して停止する。

## デフォルト観点（5 観点）

各観点は独立した `general-purpose` サブエージェントで並列起動する。各エージェントは **指定観点のみに絞ってレビューし、他観点には踏み込まない**（プロンプトで明示制約）。

### 1. 構成・主題整合性

| 項目 | 内容 |
|---|---|
| 役割 | 記事の主題（thesis）が一貫しているか、見出し構成が論理的かをチェック |
| 検出ポイント | 主題のブレ・脱線セクション・見出し階層の論理破綻・「1 記事 1 テーマ」原則違反・結論ファースト原則違反 |
| プロンプト制約 | 冗長性・読者視点・文体・ガイドライン準拠の指摘は出さない（他観点エージェントが担当する） |

### 2. 冗長性・重複・無駄な前置き

| 項目 | 内容 |
|---|---|
| 役割 | 同じ内容の繰り返し・無情報の前置き・重複セクションを検出 |
| 検出ポイント | 直前の節と内容が重複する記述・「最小実装は次の通り」のような無情報前置き・節見出しと内容が重複する宣言文 |
| プロンプト制約 | 構成・読者視点・文体・ガイドライン準拠の指摘は出さない |

### 3. 読者視点での理解しやすさ

| 項目 | 内容 |
|---|---|
| 役割 | 読者が前提知識なしで読み進められるか、用語・コード例の文脈設計が十分か |
| 検出ポイント | 用語の初出未定義・コードブロック直前の文脈設計欠如・読者の前提知識を超えた説明・図表の説明不足 |
| プロンプト制約 | 構成・冗長性・文体・ガイドライン準拠の指摘は出さない |

### 4. 文体一貫性

| 項目 | 内容 |
|---|---|
| 役割 | です/ます調と常体の混在・タイトル主体動詞と本文の混在・一人称の使い方の揺れを検出 |
| 検出ポイント | 文体の混在・ヘッジ表現（「かもしれません」「と思われます」）の濫用・物語調と仕様調の混在・括弧書きの多用 |
| プロンプト制約 | 構成・冗長性・読者視点・ガイドライン準拠の指摘は出さない |

### 5. ガイドライン準拠チェック

| 項目 | 内容 |
|---|---|
| 役割 | 対象パスに応じた `quality-guidelines.md` の各節と記事本文を機械的に照合し、違反を検出 |
| 検出ポイント | ガイドラインの「禁止」「避ける」「必須」と明記された項目に対する違反（特に「個人・環境固有情報の混入禁止」「内容の正確性」「主題（thesis）の扱い」等の節タイトルで明示禁止されている領域） |
| 参照ガイドライン | 対象ファイルパスに応じて切り替え。詳細は「処理手順 3. ガイドライン解決」を参照 |
| プロンプト制約 | 構成・冗長性・読者視点・文体の指摘は出さない（他観点エージェントが担当する）。あくまで「ガイドラインの該当節とどう違反しているか」の機械的照合で指摘する |

### 観点の独立性に関する設計意図

既存 4 観点（症状ベース）はガイドライン違反でも症状として現れない違反を検出できないことが Issue #9（loop 記事執筆時）で判明した。「個人・環境固有情報の混入禁止」節で明示禁止されている `/auto-finalize` 等のプロジェクト固有スキル名を本文に書いた違反を、4 観点はいずれも素通りした。ガイドライン準拠チェック観点を独立に並列起動することで、症状として現れない違反も検出する。

## 処理手順

### 1. 引数バリデーション

`$ARGUMENTS` を引用符を尊重した形でトークン分割し、第 1 トークンを対象ファイルパス、第 2 トークンを観点リスト（任意）として取り出す。

- 引数文字列に `"..."` または `'...'` で囲まれた区間がある場合、その区間内のスペースは区切り文字として扱わない（パスにスペースが含まれるケース対応）。引用符自体は値から除去する
- 引用符を含まない場合は単純に空白区切り
- 引用符のエスケープ（`\"` 等）は想定しない（記事ファイルパスにエスケープ引用符が含まれる用途はないため）。閉じられない引用符はエラー表示して停止する
- 第 1 トークンが空の場合はエラー表示して停止する
- **第 3 トークン以降が存在する場合**はエラー表示して停止する（誤入力の黙殺を防ぐ）。観点を複数指定する場合は第 2 トークン内でカンマ区切りで連結する（例: `foo.md 文体,構成` は OK、`foo.md 文体 構成` はエラー）
- `Bash` ツールで `[ -f "<対象ファイルパス>" ]` を実行してファイル存在を確認する。存在しない場合はエラー表示して停止する
- 拡張子が `.md` 以外の場合はエラー表示して停止する（適用範囲: 記事 markdown 専用）
- パスが `articles/` 配下でない場合は警告を表示し、ユーザーに「対象が記事ファイルか」を確認してから続行する。仕様書・README 等の非記事文書を誤って渡すと、観点プロンプト（記事向けに調整済）と検出ルールが噛み合わず誤検出が発生するため

例:

- `articles/zenn/foo.md` → 対象パス: `articles/zenn/foo.md`、観点: 省略
- `"C:/My Articles/foo.md" 構成,読者視点` → 対象パス: `C:/My Articles/foo.md`、観点: `構成,読者視点`
- `articles/zenn/foo.md 文体` → 対象パス: `articles/zenn/foo.md`、観点: `文体`

### 2. 観点リストの確定

第 2 引数が省略された場合はデフォルト 5 観点全件を使用する。指定がある場合はカンマ区切りで分解する。

| 識別子 | 観点 |
|---|---|
| `構成` または `構成・主題整合性` | 1. 構成・主題整合性 |
| `冗長性` または `冗長性・重複` | 2. 冗長性・重複・無駄な前置き |
| `読者視点` | 3. 読者視点での理解しやすさ |
| `文体` または `文体一貫性` | 4. 文体一貫性 |
| `ガイドライン` または `ガイドライン準拠` | 5. ガイドライン準拠チェック |

省略形・正式名いずれも受け付ける。未知の識別子はエラー表示して停止する。

**重複の正規化**: 識別子を上記テーブルで観点番号（1〜5）に解決した後、**重複は除外して一意化** する。同じ観点を指す識別子（例: `構成,構成`、`構成,構成・主題整合性`）が含まれる場合は重複を取り除いてから並列起動する。トークン消費・結果重複の防止が目的。

### 3. ガイドライン解決（観点 5 を含む場合のみ）

確定観点に観点 5（ガイドライン準拠チェック）が含まれる場合、対象ファイルパスから参照すべき `quality-guidelines.md` を解決する。

| 対象パスの前方一致 | 参照ガイドラインファイル |
|---|---|
| `articles/zenn/` | `.claude/skills/write-zenn/quality-guidelines.md` |
| `articles/hatena/` | `.claude/skills/write-hatena-diary/quality-guidelines.md` |
| 上記いずれにもマッチしない | 観点 5 のみエラー停止: `エラー: ガイドライン参照先が解決できない対象パスです: <TARGET_FILE>` |

**対象パスの形式**: 本テーブルは **リポルートからの相対パス** に対する前方一致で判定する。絶対パス（`C:/.../articles/hatena/foo.md` 等）は本 MVP では未サポート。観点 5 がパス解決できずエラー停止する。絶対パスを使う場合は呼び出し側で相対パスに変換してから渡すこと（将来サポートが必要なら別 Issue で対応）。

将来別媒体が追加された場合は本テーブルにエントリを追加する。

観点 5 がエラー停止した場合でも、観点 1〜4 は引き続き並列起動する（部分的成功を許容）。観点 5 のエラーは最終結果セクションに警告として含める。

### 4. サブエージェント並列起動

確定した観点ごとに `Task` ツールで `subagent_type=general-purpose` を **観点数だけ並列起動** する。1 メッセージ内で複数の Task tool use を含めることで並列実行する。

各エージェントへのプロンプトは以下の構造で構築する。`## 参照ガイドライン` セクションは **観点 5（ガイドライン準拠チェック）を起動する場合のみ** プロンプトに含める（観点 1〜4 では本セクションを省略）。観点 5 のプロンプトでは、ステップ 3 で解決した `quality-guidelines.md` のパスを記載する。

```text
あなたは技術記事 markdown を「<観点名>」観点のみでレビューするエージェントです。

## 対象ファイル
<TARGET_FILE の絶対パス>

## あなたの担当観点
<観点 N の役割・検出ポイント・プロンプト制約をそのままコピー>

## 参照ガイドライン
<ステップ 3 で解決した quality-guidelines.md のパス>

## 制約
- 担当観点の指摘のみ出してください。他観点（<他観点リスト>）には踏み込まないでください
- 指摘がない場合は「指摘なし」と返してください
- 推測や曖昧表現を避け、具体的な行番号・該当箇所を引用してください

## 出力フォーマット
以下の markdown セクション形式で返してください:

### <観点名>

- **指摘 1**: <該当箇所（行番号 or 引用）>
  - 問題: <何が問題か>
  - 提案: <どう直すか>
- **指摘 2**: ...

(指摘がない場合は「指摘なし」のみ)
```

### 5. 結果集約

並列起動した全サブエージェントの結果を回収し、観点順に連結する。
ステップ 3 で観点 5 がエラー停止した場合は、結果セクション末尾に警告として「ガイドライン準拠チェックは参照先未解決のためスキップしました（対象パス: `<TARGET_FILE>`）」を追記する。

出力先パスの組み立て:

- ディレクトリ: `.tmp/multi-perspective-review/`（存在しなければ `Bash` ツールで `mkdir -p` で作成）
- ファイル名: `<日時>-<slug>.md`
  - `<日時>`: `YYYYMMDD-HHMMSS` 形式（`Bash` ツールで `date '+%Y%m%d-%H%M%S'` 等で取得）
  - `<slug>`: 対象ファイル名から拡張子を除いた値（例: `articles/zenn/20260505-foo.md` → `20260505-foo`）

集約ファイルのフォーマット:

```markdown
# 観点別レビュー結果

- 対象ファイル: `<TARGET_FILE>`
- 実行日時: <ISO8601>
- 実行観点: <観点名のリスト>

## サマリ

| 観点 | 指摘件数 |
|---|---|
| <実行した観点 1 名> | N |
| <実行した観点 2 名> | N |
| ... | ... |

## 詳細

<各観点エージェントの出力（### 見出し付きセクション）をそのまま連結>
```

サマリ表のルール:

- **行構成は実行した観点のみ**。観点絞り実行時（例: `文体,読者視点` のみ）は該当 2 行のみを表示する。未実行観点はサマリ表に含めない（誤解を防ぐ）
- 件数列の値の決め方（判定順序: エラー判定 → 正常応答内のカウント）:
  - サブエージェント実行が失敗した観点 → `エラー`（数値ではなく文字列で記録）。合計件数の計算からは除外し、合計表示の末尾に `（エラー: M 観点）` を付記する
  - サブエージェントが正常応答した観点 → 出力中に出現する `**指摘` の回数を `Bash` ツール経由の grep でカウント（書式揺れに耐えるため厳密な `**指摘 N**:` ではなく `**指摘` の前方一致で数える）。「指摘なし」のみの応答は `**指摘` を含まないため自動的に 0 件となる

### 6. 結果の自動オープン

`code -g "<OUTPUT_FILE の絶対パス>:1"` で集約ファイルをエディタで自動オープンする。

`code` コマンドが PATH に通っていない場合は警告のみ出して続行する（出力ファイル自体は生成済みのため、ユーザーが手動でファイルを開けば確認可能）。

### 7. チャット表示

チャットには **ファイルパスと観点別件数のみ** を表示する。詳細はファイル側で確認する設計。観点別件数の行はサマリ表と同じく **実行した観点のみ** を列挙する。

```text
✅ /multi-perspective-review 完了

📁 出力: .tmp/multi-perspective-review/<TIMESTAMP>-<SLUG>.md

📊 観点別指摘件数:
  - <実行した観点 1 名>: N 件
  - <実行した観点 2 名>: N 件
  ...

合計: N 件（エディタで開きました）
```

表示ルール:

- **観点絞り実行時**: 指定された観点の行のみを列挙する（未実行観点の行は出さない）
- **エラー観点がある場合**: 該当観点は `<観点名>: エラー（<理由要約>）` と表示し、合計件数の計算には含めない。合計表示の末尾に `（エラー: M 観点）` を付記する
- **`code` 起動が成功しなかった場合**（未インストール / 起動失敗の双方を含む）: 末尾の `（エディタで開きました）` を `（手動でファイルを開いてください）` に置き換える。`code` の exit status が 0 の場合のみ「エディタで開きました」と表示する

## 起動シーン

- **記事生成完了後・公開前の最終確認**: `/write-zenn` または `/write-hatena-diary` で記事を生成した後、公開前に任意のタイミングで起動する
- **既存記事の改善時**: 記事の改稿・追記後に観点を絞って再レビュー（例: 文体だけ確認したい場合は `/multi-perspective-review <パス> 文体`）

各記事生成スキル（`/write-zenn` / `/write-hatena-diary`）からの自動呼び出しはしない（ユーザーが必要時に明示的に呼ぶ）。

## エラーハンドリング

| 状況 | 対応 |
|---|---|
| 引数なし | エラー表示して停止 |
| 対象ファイルが存在しない | エラー表示して停止 |
| 観点引数に未知の識別子 | エラー表示して停止（受付可能な識別子一覧を併記） |
| サブエージェント実行失敗（一部） | 失敗した観点をサマリに `エラー` として記録し、他観点の結果は継続して集約 |
| 観点 5 のガイドライン参照先が解決できない | 観点 5 のみエラー停止、観点 1〜4 は継続。最終結果に警告として記載 |
| 全サブエージェント失敗 | エラー表示して停止 |
| `code` コマンド未インストール | 警告表示のみで続行（出力ファイルは生成済み） |
| `.tmp/` ディレクトリ作成失敗 | エラー表示して停止 |
