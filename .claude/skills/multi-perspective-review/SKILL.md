## タスク

技術記事 markdown の品質を、観点別に独立した `general-purpose` サブエージェントで並列レビューする。各エージェントが指定観点のみに絞ることで、観点が交差せず重複のない指摘を得る。

`doc-reviewer` エージェントは仕様書レビュー専用の観点を持っており、記事には合わない。本スキルは記事 markdown 専用の汎用観点レビュー手段として機能する。

## 適用範囲

- **対象**: 記事 markdown（`articles/zenn/*.md` を主想定。将来別媒体（Qiita, note 等）の記事 markdown が追加された場合も、本 skill を流用できる設計とする。媒体ごとの専用 skill 化はしない方針）
- **対象外**: 仕様書・README・コード等の非記事文書（`doc-reviewer` / `code-reviewer` を使用）

## 引数

`$ARGUMENTS` の形式:

- **`<対象ファイルパス>`**: 必須。レビュー対象の記事 markdown ファイルへのパス（リポジトリルートからの相対 or 絶対）
- **`<対象ファイルパス> [観点1,観点2,...]`**: 第 2 引数（観点リスト）は任意。指定時は **カンマ区切り**（スペース不可）で渡す。省略時はデフォルト 5 観点全件で並列起動

例:

- `/multi-perspective-review articles/zenn/20260505-xxx.md`
- `/multi-perspective-review articles/zenn/20260505-xxx.md 構成,読者視点`
- `/multi-perspective-review articles/zenn/20260505-xxx.md 文体`（単一観点はカンマ不要）

第 2 引数全体が 1 トークン。観点識別子に含まれる `・` は区切り文字として扱わない（`構成・主題整合性,読者視点` のように正式名 + カンマ区切りも許容）。

引数なし、または 1 つ目の引数がファイルとして存在しない場合はエラーを表示して停止する。

## デフォルト観点（5 観点）

各観点は独立した `general-purpose` サブエージェントで並列起動する。各エージェントは **指定観点のみに絞ってレビューし、他観点には踏み込まない**（プロンプトで明示制約）。

### 1. 構成・主題整合性

| 項目 | 内容 |
|---|---|
| 役割 | 記事の主題（thesis）が一貫しているか、見出し構成が論理的かをチェック |
| 検出ポイント | 主題のブレ・脱線セクション・見出し階層の論理破綻・「1 記事 1 テーマ」原則違反・結論ファースト原則違反 |
| プロンプト制約 | 冗長性・読者視点・文体・ガイドライン準拠の指摘は出さない（他観点エージェントが担当する） |

### 2. 冗長性・重複・無駄な前置き

| 項目 | 内容 |
|---|---|
| 役割 | 同じ内容の繰り返し・無情報の前置き・重複セクションを検出 |
| 検出ポイント | 直前の節と内容が重複する記述・「最小実装は次の通り」のような無情報前置き・節見出しと内容が重複する宣言文 |
| プロンプト制約 | 構成・読者視点・文体・ガイドライン準拠の指摘は出さない |

### 3. 読者視点での理解しやすさ

| 項目 | 内容 |
|---|---|
| 役割 | 読者が前提知識なしで読み進められるか、用語・コード例の文脈設計が十分か |
| 検出ポイント | 用語の初出未定義・コードブロック直前の文脈設計欠如・読者の前提知識を超えた説明・図表の説明不足 |
| プロンプト制約 | 構成・冗長性・文体・ガイドライン準拠の指摘は出さない |

### 4. 文体一貫性

| 項目 | 内容 |
|---|---|
| 役割 | です/ます調と常体の混在・タイトル主体動詞と本文の混在・一人称の使い方の揺れを検出 |
| 検出ポイント | 文体の混在・ヘッジ表現（「かもしれません」「と思われます」）の濫用・物語調と仕様調の混在・括弧書きの多用 |
| プロンプト制約 | 構成・冗長性・読者視点・ガイドライン準拠の指摘は出さない |

### 5. ガイドライン準拠チェック

| 項目 | 内容 |
|---|---|
| 役割 | `.claude/skills/write-zenn/quality-guidelines.md` の各節と記事本文を機械的に照合し、違反を検出 |
| 検出ポイント | ガイドラインの「禁止」「避ける」「必須」と明記された項目に対する違反（特に「個人・環境固有情報の混入禁止」「内容の正確性」「主題（thesis）の扱い」等の節タイトルで明示禁止されている領域） |
| 参照ガイドライン | `.claude/skills/write-zenn/quality-guidelines.md`（固定参照） |
| プロンプト制約 | 構成・冗長性・読者視点・文体の指摘は出さない（他観点エージェントが担当する）。あくまで「ガイドラインの該当節とどう違反しているか」の機械的照合で指摘する |

### 観点の独立性に関する設計意図

既存 4 観点（症状ベース）はガイドライン違反でも症状として現れない違反を検出できないことが Issue #9（loop 記事執筆時）で判明した。「個人・環境固有情報の混入禁止」節で明示禁止されている `/auto-finalize` 等のプロジェクト固有スキル名を本文に書いた違反を、4 観点はいずれも素通りした。ガイドライン準拠チェック観点を独立に並列起動することで、症状として現れない違反も検出する。

## 処理手順

### 1. 引数バリデーション

`$ARGUMENTS` を空白で分割し、先頭トークンを対象ファイルパス、2 トークン目を観点リスト（任意）として取り出す。

- 先頭トークンが空の場合はエラー表示して停止する
- `Bash` ツールで `[ -f "<対象ファイルパス>" ]` を実行してファイル存在を確認する。存在しない場合はエラー表示して停止する

### 2. 観点リストの確定

第 2 引数が省略された場合はデフォルト 5 観点全件を使用する。指定がある場合はカンマ区切りで分解する。

| 識別子 | 観点 |
|---|---|
| `構成` または `構成・主題整合性` | 1. 構成・主題整合性 |
| `冗長性` または `冗長性・重複` | 2. 冗長性・重複・無駄な前置き |
| `読者視点` | 3. 読者視点での理解しやすさ |
| `文体` または `文体一貫性` | 4. 文体一貫性 |
| `ガイドライン` または `ガイドライン準拠` | 5. ガイドライン準拠チェック |

省略形・正式名いずれも受け付ける。未知の識別子はエラー表示して停止する。

### 3. サブエージェント並列起動

確定した観点ごとに `Task` ツールで `subagent_type=general-purpose` を **観点数だけ並列起動** する。1 メッセージ内で複数の Task tool use を含めることで並列実行する。

各エージェントへのプロンプトは以下の構造で構築する。`## 参照ガイドライン` セクションは **観点 5（ガイドライン準拠チェック）を起動する場合のみ** プロンプトに含める。観点 1〜4 では本セクションを省略する。

```text
あなたは技術記事 markdown を「<観点名>」観点のみでレビューするエージェントです。

## 対象ファイル
<TARGET_FILE の絶対パス>

## あなたの担当観点
<観点 N の役割・検出ポイント・プロンプト制約をそのままコピー>

## 参照ガイドライン
.claude/skills/write-zenn/quality-guidelines.md

## 制約
- 担当観点の指摘のみ出してください。他観点（<他観点リスト>）には踏み込まないでください
- 指摘がない場合は「指摘なし」と返してください
- 推測や曖昧表現を避け、具体的な行番号・該当箇所を引用してください

## 出力フォーマット
以下の markdown セクション形式で返してください:

### <観点名>

- **指摘 1**: <該当箇所（行番号 or 引用）>
  - 問題: <何が問題か>
  - 提案: <どう直すか>
- **指摘 2**: ...

(指摘がない場合は「指摘なし」のみ)
```

### 4. 結果集約

並列起動した全サブエージェントの結果を回収し、観点順に連結する。

出力先パスの組み立て:

- ディレクトリ: `.tmp/multi-perspective-review/`（存在しなければ `Bash` ツールで `mkdir -p` で作成）
- ファイル名: `<日時>-<slug>.md`
  - `<日時>`: `YYYYMMDD-HHMMSS` 形式（`Bash` ツールで `date '+%Y%m%d-%H%M%S'` 等で取得）
  - `<slug>`: 対象ファイル名から拡張子を除いた値（例: `articles/zenn/20260505-foo.md` → `20260505-foo`）

集約ファイルのフォーマット:

```markdown
# 観点別レビュー結果

- 対象ファイル: `<TARGET_FILE>`
- 実行日時: <ISO8601>
- 実行観点: <観点名のリスト>

## サマリ

| 観点 | 指摘件数 |
|---|---|
| 構成・主題整合性 | N |
| 冗長性・重複・無駄な前置き | N |
| 読者視点での理解しやすさ | N |
| 文体一貫性 | N |
| ガイドライン準拠チェック | N |

## 詳細

<各観点エージェントの出力（### 見出し付きセクション）をそのまま連結>
```

サマリの「指摘件数」は各エージェントの出力から指摘数をカウントする。

- 「指摘なし」のみが返ってきた観点は 0 件
- それ以外は出力中に出現する `**指摘` の回数を `Bash` ツール経由の grep でカウントする（書式揺れに耐えるため厳密な `**指摘 N**:` ではなく `**指摘` の前方一致で数える）

### 5. 結果の自動オープン

`code -g "<OUTPUT_FILE の絶対パス>:1"` で集約ファイルをエディタで自動オープンする。

`code` コマンドが PATH に通っていない場合は警告のみ出して続行する（出力ファイル自体は生成済みのため、ユーザーが手動でファイルを開けば確認可能）。

### 6. チャット表示

チャットには **ファイルパスと観点別件数のみ** を表示する。詳細はファイル側で確認する設計。

```text
✅ /multi-perspective-review 完了

📁 出力: .tmp/multi-perspective-review/<TIMESTAMP>-<SLUG>.md

📊 観点別指摘件数:
  - 構成・主題整合性: N 件
  - 冗長性・重複: N 件
  - 読者視点: N 件
  - 文体一貫性: N 件
  - ガイドライン準拠: N 件

合計: N 件（エディタで開きました）
```

## 起動シーン

- **記事生成完了後・公開前の最終確認**: `/write-zenn` で記事を生成した後、公開前に任意のタイミングで起動する
- **既存記事の改善時**: 記事の改稿・追記後に観点を絞って再レビュー（例: 文体だけ確認したい場合は `/multi-perspective-review <パス> 文体`）

`/write-zenn` からの自動呼び出しはしない（ユーザーが必要時に明示的に呼ぶ）。

## エラーハンドリング

| 状況 | 対応 |
|---|---|
| 引数なし | エラー表示して停止 |
| 対象ファイルが存在しない | エラー表示して停止 |
| 観点引数に未知の識別子 | エラー表示して停止（受付可能な識別子一覧を併記） |
| サブエージェント実行失敗（一部） | 失敗した観点をサマリに `エラー` として記録し、他観点の結果は継続して集約 |
| 全サブエージェント失敗 | エラー表示して停止 |
| `code` コマンド未インストール | 警告表示のみで続行（出力ファイルは生成済み） |
| `.tmp/` ディレクトリ作成失敗 | エラー表示して停止 |
