## タスク

はてなブログ向け日記記事（`articles/hatena/*.md`）を、日記特性に絞った 3 観点で独立サブエージェントが並列レビューする。レビュー結果の指摘は `/triage` 経由でユーザーに確認し、確定した修正方針を対象記事ファイルへ Edit で適用する。

本スキルは Zenn 向け技術記事を対象とする `/multi-perspective-review` とは別系統。日記の対話形式特有の品質ルールに特化しており、技術記事向けの観点（コードブロック前提の読者視点・thesis 整合性等）は持たない。

## 起動方法

- **自動呼び出し**: `/write-hatena-diary` が各記事生成直後（Phase 5 直後・ループ内）に本スキルを実行する
- **手動起動**: ユーザーが任意のタイミングで `/review-hatena-diary <パス>` を実行する

両起動経路で処理内容は同一。

## 引数

`$ARGUMENTS` の形式:

- **`<対象ファイルパス>`**: 必須。レビュー対象の日記記事 markdown ファイルへのパス。**リポジトリルートからの相対パス必須**（絶対パスはエラー停止する。観点プロンプトと参照ガイドライン解決の安定性のため）

例:

- `/review-hatena-diary articles/hatena/2026-05-16-11-46-00-diary-style-revamp.md`

引数が空、または対象ファイルが存在しない場合はエラー表示して停止する。

## 観点（3 観点固定）

3 観点は固定。観点引数による絞り込みはサポートしない（最低限のチェックに集中する設計判断）。各観点は独立した `general-purpose` サブエージェントで **並列起動** する。各エージェントは **指定観点のみに絞ってレビューし、他観点には踏み込まない**（プロンプトで明示制約）。

### 1. キャラ整合性

| 項目 | 内容 |
|---|---|
| 役割 | キャラ設定（一人称・複数形・口調・呼称）の取り違えを検出する |
| 検出ポイント | クロちゃんが「私」「私たち」を使う / 幸田姉さんが「ぼく」「ぼくら」を使う / 幸田姉さんがクロちゃんを「あんた」以外で呼ぶ / 記事冒頭以外で「幸田姉さん」フル表記が使われる / 名前由来やキャラ設定が会話内に混入する / 社長が話者として対話に登場する |
| 参照ファイル | `.claude/skills/write-hatena-diary/quality-guidelines.md` Part 1「登場人物」「2 人の関係」「オーナー（社長）の位置づけ」 |
| プロンプト制約 | ガイドライン準拠・言及正確性の指摘は出さない（他観点エージェントが担当する）。Part 1（物語世界）の遵守チェックに専念する |

### 2. ガイドライン準拠

| 項目 | 内容 |
|---|---|
| 役割 | `quality-guidelines.md` Part 2「品質ルール」の各節と記事本文を機械的に照合し、違反を検出する |
| 検出ポイント | タイトル方針違反 / 1 H2 1 話題違反 / 1 セリフ 1 論点違反 / 内部実装語の過剰表記 / カッコ書きの使用 / 簡素記法の不備（マーカー残存・閉じ忘れ・入れ子）/ リポジトリ参照ルール違反（`becky3/<name>` 形式 / GitHub URL / Issue 番号の本文混入）/ 個人・環境固有情報の混入 / 著作物・IP 情報の混入（実在作品由来の決め台詞・固有口調等の暗黙引用を含む） |
| 参照ファイル | `.claude/skills/write-hatena-diary/quality-guidelines.md` Part 2 全節 + `.claude/skills/write-hatena-diary/template-diary.md` + `.claude/skills/write-hatena-diary/balloon-html.md` + `~/.claude/rules/invariants.md`「著作物・IP 情報の混入防止」 |
| プロンプト制約 | キャラ整合性・言及正確性の指摘は出さない。Part 2 機械照合と著作物・IP 検出に専念する |

### 3. 言及正確性

| 項目 | 内容 |
|---|---|
| 役割 | 記事本文の事実関係（リポ名・スキル名・固有名詞・ジャーナルから派生した出来事の描写）が素材ジャーナルと整合しているかを検出する |
| 検出ポイント | 存在しないリポ名・スキル名の言及 / リポ名のタイポ / ジャーナルに記載のない出来事を事実として書く / 推測・憶測を事実として書く / 数値・日付・固有名詞の誤り |
| 参照ファイル | 対象記事のフロントマター `date:` の日付に紐づくジャーナル本文（処理手順 2 で取得・サブエージェントへ渡す） + リポジトリマスターテーブル（`.claude/skills/write-hatena-diary/template-diary.md`） |
| プロンプト制約 | キャラ整合性・ガイドライン準拠の指摘は出さない。素材ジャーナルとの照合による事実誤認の検出に専念する |

### 観点設計の意図

3 観点に絞ることで、サブエージェントの実行コストとユーザーが triage で確認する負荷を最小化する。物語性の高い日記に対してレビュー観点が多すぎると、強制力が強くなり面白さが落ちるリスクがあるため、「明らかな間違い・禁止事項・発言ミス」に集中する設計判断（Issue #48 第 1 回 triage Q3）。

## 処理手順

### 1. 引数バリデーション

`$ARGUMENTS` を空白で分割し、最初の非空トークンを対象ファイルパス `TARGET_FILE` とする。

- 第 1 トークンが空の場合: エラー停止: `エラー: 対象ファイルパスを指定してください`
- 絶対パス（以下のいずれかにマッチする場合）はエラー停止: `エラー: 対象ファイルパスはリポジトリルートからの相対パスで指定してください: <TARGET_FILE>`
  - 正規表現 `^[A-Za-z]:[/\\]`（Windows ドライブレター付き絶対パス、例: `C:/foo`、`C:\foo`）
  - 正規表現 `^/`（Unix 絶対パス、例: `/home/foo`）
  - 正規表現 `^\\\\`（Windows UNC パス、例: `\\server\share`）
- `Bash` ツールで `[ -f "$TARGET_FILE" ]` を実行してファイル存在を確認する。存在しない場合: エラー停止: `エラー: 対象ファイルが見つかりません: <TARGET_FILE>`
- 拡張子が `.md` 以外の場合: エラー停止: `エラー: 対象ファイルは Markdown (.md) を指定してください: <TARGET_FILE>`
- パスが `articles/hatena/` 配下でない場合: エラー停止: `エラー: 本スキルは articles/hatena/ 配下の日記記事を対象としています: <TARGET_FILE>`

### 2. 素材取得（言及正確性観点向け）

観点 3「言及正確性」のサブエージェントに渡すジャーナル素材を取得する。

なお、後続のサブエージェントプロンプトには `TARGET_FILE` の絶対パスを埋め込む必要があるため、本ステップ冒頭で `TARGET_FILE_ABS = "$(pwd)/$TARGET_FILE"` を構築しておく（リポジトリルートで実行されている前提）。

1. `Read` で `TARGET_FILE` を読み、フロントマター `date:` から記事の日記対象日 `ARTICLE_DATE` を抽出する
2. フロントマター `date:` が存在しない・形式が不正な場合: 警告を表示し、観点 3「言及正確性」は **ジャーナル照合を行わずリポジトリマスターテーブルのみ参照する** モードで起動する（観点 1・2 は通常通り起動）
3. `rag_list_by_date_range(date_from=ARTICLE_DATE, date_to=ARTICLE_DATE, source_type="journal", limit=100)` で当日のジャーナル一覧を取得
4. 各ジャーナルについて `rag_get_document(source_id=...)` で全文取得し、`JOURNAL_TEXTS` として保持する。個別取得失敗時は警告を表示し該当ジャーナルをスキップ、取得済みのみで観点 3 を実行する（全件失敗時はステップ 5 と同じくリポマスターのみ参照モードに降格）
5. ジャーナルが 0 件の場合: 警告を表示し、観点 3 はリポジトリマスターテーブルのみ参照するモードで起動する

### 3. サブエージェント並列起動

3 観点について `Task` ツールで `subagent_type=general-purpose` を **3 並列起動** する。1 メッセージ内で 3 つの Task tool use を含めることで並列実行する。

各エージェントへのプロンプトは以下の構造で構築する:

```text
あなたははてなブログ向け日記記事を「<観点名>」観点のみでレビューするエージェントです。

## 対象ファイル
<TARGET_FILE_ABS（ステップ 2 で構築した絶対パス）>

## あなたの担当観点
<観点 N の役割・検出ポイント・プロンプト制約をそのままコピー>

## 参照ファイル
<観点 N の参照ファイル一覧（絶対パス）>

## 素材ジャーナル（観点 3「言及正確性」のみ）
<JOURNAL_TEXTS を埋め込む。記事本文の事実関係照合に使用する>

## 制約
- 担当観点の指摘のみ出してください。他観点（<他観点リスト>）には踏み込まないでください
- 指摘がない場合は「指摘なし」と返してください
- 推測や曖昧表現を避け、具体的な行番号・該当箇所を引用してください

## 出力フォーマット
以下の markdown セクション形式で返してください。**「提案: 」プレフィックスは厳守**（後続の triage 連携で文字列マッチングに利用するため、「修正案」「修正提案」等への言い換えは不可）:

### <観点名>

- **指摘 1**: <該当箇所（行番号 or 引用）>
  - 問題: <何が問題か>
  - 提案: <どう直すか（具体的な置換文字列・diff・修正方針を明示）>
- **指摘 2**: ...

(指摘がない場合は「指摘なし」のみ)
```

### 4. 結果集約 + ファイル出力

並列起動した 3 サブエージェントの結果を回収し、観点順に連結する。

出力先パスの組み立て:

- ディレクトリ: `.tmp/review-hatena-diary/`（存在しなければ `Bash` ツールで `mkdir -p` で作成）
- ファイル名: `<日時>-<slug>.md`
  - `<日時>`: `YYYYMMDD-HHMMSS` 形式（`Bash` ツールで `date '+%Y%m%d-%H%M%S'` で取得）
  - `<slug>`: 対象ファイル名から拡張子を除いた値（例: `articles/hatena/2026-05-16-11-46-00-foo.md` → `2026-05-16-11-46-00-foo`）

集約ファイルのフォーマット:

```markdown
# 日記レビュー結果

- 対象ファイル: `<TARGET_FILE>`
- 実行日時: <ISO8601>
- 参照ジャーナル件数: N 件（観点 3 用）

## サマリ

| 観点 | 指摘件数 |
|---|---|
| キャラ整合性 | N |
| ガイドライン準拠 | N |
| 言及正確性 | N |

## 詳細

<各観点エージェントの出力（### 見出し付きセクション）をそのまま連結>
```

サマリ表の件数列の値:

- サブエージェント実行が失敗した観点 → `エラー`（数値ではなく文字列で記録）。合計件数の計算からは除外
- サブエージェントが正常応答した観点 → 集約ファイル生成後、当該観点の `###` 見出しから次の `###` 見出しまでの範囲内で `- **指摘` 始まりの行を awk で範囲抽出してカウントする。観点をまたいだ誤集計を防ぐため、必ず当該観点の見出し範囲内に絞る:

  ```bash
  awk '/^### キャラ整合性$/{f=1;next} /^### /{f=0} f && /^- \*\*指摘/' <OUTPUT_FILE> | wc -l
  ```

### 5. 結果の自動オープン

`code -g "<OUTPUT_FILE の絶対パス>:1"` で集約ファイルをエディタで自動オープンする。

`code` コマンドが PATH に通っていない場合は警告のみ出して続行する（出力ファイル自体は生成済みのため、ユーザーが手動でファイルを開けば確認可能）。

**リモートセッション時の動作**: 環境変数 `REMOTE_SLACK_SESSION=1` が設定されている場合は `code` 起動を行わず、出力ファイルパスをチャットに表示する代替動線に切り替える（`~/.claude/CLAUDE.md`「リモートセッション時の確認動線切替」準拠）。

### 6. 指摘の triage 連携

集約ファイルから全指摘を抽出し、`/triage` スキルを `Skill` ツールで起動する。triage の分類・確認手段の選択・修正方針確定の動作仕様は `~/.claude/skills/triage/SKILL.md` を SSoT として参照する（本スキル内では分類軸を断定しない）。

triage への引数には以下を含める:

- 各指摘の対象箇所（ファイル名:行番号）
- 各指摘の要約
- 各指摘の「提案」（観点プロンプトの出力フォーマット「提案: 」フィールド）
- 全指摘の対象ファイルが `TARGET_FILE` であること

triage が中断（ユーザーが途中で打ち切り）された場合: それまでに確定した修正のみ次ステップで適用する。確定していない指摘はスキップ扱い。

### 7. 修正適用

triage で「修正する」または「別案がある」と確定した指摘について、`Edit` ツールで `TARGET_FILE` に修正を適用する。

- 「修正する」: 観点エージェントの「提案」を採用
- 「別案がある」: ユーザーが triage で提示した別案を採用

修正適用は 1 指摘ずつ Edit を呼び出す（複数指摘を 1 Edit でまとめて適用しない）。以下の場合は **適用失敗として警告を表示しスキップ**、残りの指摘の修正を続行する:

- 観点エージェントの「提案」が `old_string` / `new_string` 形式で揃わない・整形不能（自由形式の修正方針のみで具体的な置換文字列が抽出できない場合等。本スキルでは無理に整形しない）
- `Edit` 実行時に `old_string` が一意でない・存在しない等の理由で失敗する

修正適用後の **再レビューは行わない**。1 回の review → triage → 修正で完結する。

### 8. チャット表示

チャットには **ファイルパス・観点別件数・修正適用結果** を表示する。詳細は集約ファイル側で確認する設計。

```text
✅ /review-hatena-diary 完了

📁 対象記事: <TARGET_FILE>
📁 レビュー結果: .tmp/review-hatena-diary/<TIMESTAMP>-<SLUG>.md

📊 観点別指摘件数:
  - キャラ整合性: N 件
  - ガイドライン準拠: N 件
  - 言及正確性: N 件

🛠️ 修正適用:
  - 適用: N 件
  - スキップ: N 件
  - 適用失敗: N 件

合計: N 件（エディタで開きました）
```

表示ルール:

- **エラー観点がある場合**: 該当観点は `<観点名>: エラー（<理由要約>）` と表示し、合計件数の計算には含めない。合計表示の末尾に `（エラー: M 観点）` を付記する
- **`code` 起動が成功しなかった場合**: 末尾の `（エディタで開きました）` を `（手動でファイルを開いてください）` に置き換える
- **修正適用失敗が 0 件の場合**: `適用失敗: 0 件` の行は省略してよい

## エラーハンドリング

| 状況 | 対応 |
|---|---|
| 引数なし | エラー表示して停止 |
| 絶対パス指定 | エラー表示して停止 |
| 対象ファイルが存在しない | エラー表示して停止 |
| 拡張子が `.md` でない | エラー表示して停止 |
| `articles/hatena/` 配下でない | エラー表示して停止 |
| フロントマター `date:` 不在・不正 | 警告表示 + 観点 3 をジャーナル照合なしモードで起動 |
| `rag_list_by_date_range` 失敗 | 警告表示 + 観点 3 をジャーナル照合なしモードで起動（観点 1・2 は継続） |
| 当日ジャーナル 0 件 | 警告表示 + 観点 3 をリポマスターのみ参照モードで起動 |
| サブエージェント実行失敗（一部） | 失敗した観点をサマリに `エラー` として記録し、他観点の結果は継続して集約・triage に進む |
| 全サブエージェント失敗 | エラー表示して停止（triage 連携・修正適用はスキップ） |
| `rag_get_document` 個別失敗（一部ジャーナル取得失敗） | 警告表示 + 該当ジャーナルをスキップ、取得済みのみで観点 3 を実行。全件失敗時はリポマスターのみ参照モードに降格 |
| `code` コマンド未インストール | 警告表示のみで続行 |
| リモートセッション環境（`REMOTE_SLACK_SESSION=1`） | `code` 起動をスキップし、出力ファイルパスをチャットに表示 |
| `.tmp/` ディレクトリ作成失敗 | エラー表示して停止 |
| triage 中断（ユーザー打ち切り） | それまでに確定した修正のみ適用、未確定はスキップ扱いで続行 |
| Edit による修正適用失敗（`old_string` 不一致等） | 警告表示 + 該当指摘をスキップして残りを続行。チャット表示の「適用失敗」に件数を計上 |
