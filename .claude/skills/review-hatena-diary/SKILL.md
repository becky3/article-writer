## タスク

はてなブログ向け日記記事（`articles/hatena/*.md`）を、日記特性に絞った観点別エージェントで並列レビューする。レビュー結果の指摘は `/triage` 経由でユーザーに確認し、確定した修正方針を対象記事ファイルへ Edit で適用する。

本スキルは Zenn 向け技術記事を対象とする `/multi-perspective-review` とは別系統。日記の対話形式特有の品質ルールに特化しており、技術記事向けの観点（コードブロック前提の読者視点・thesis 整合性等）は持たない。

## 起動方法

- **自動呼び出し（通常）**: `/write-hatena-diary` が各記事生成直後（Phase 5 直後・ループ内）に本スキルを実行する
- **自動呼び出し（無人）**: `/auto-publish-diary` 経由の `/write-hatena-diary --auto-publish` から `--auto-publish` 付きで起動される
- **手動起動**: ユーザーが任意のタイミングで `/review-hatena-diary <パス>` を実行する

`--auto-publish` 指定の有無で triage 連携の挙動が分岐する。詳細は <<### 5. 指摘の自動分類@self>> 以降を参照。

## 引数

`$ARGUMENTS` の形式:

- **`<対象ファイルパス>`**: 必須。レビュー対象の日記記事 markdown ファイルへのパス。**リポジトリルートからの相対パス必須**（絶対パスはエラー停止する。観点プロンプトと参照ガイドライン解決の安定性のため）
- **`--auto-publish`**: 任意。`/auto-publish-diary` 経由で起動される無人実行モード。triage 経由のユーザー確認をスキップし、書き手が指摘ごとに採否を自己判断する。詳細は <<### 5. 指摘の自動分類@self>> 以降を参照

例:

- 通常: `/review-hatena-diary articles/hatena/2026-05-16-11-46-00-diary-style-revamp.md`
- 無人: `/review-hatena-diary articles/hatena/2026-05-16-11-46-00-diary-style-revamp.md --auto-publish`

引数が空、または対象ファイルが存在しない場合はエラー表示して停止する。

## 観点

観点は固定。観点引数による絞り込みはサポートしない。各観点は独立した `general-purpose` サブエージェントで **並列起動** する。各エージェントは **指定観点のみに絞ってレビューし、他観点には踏み込まない**（プロンプトで明示制約）。

### 1. キャラ整合性

| 項目 | 内容 |
|---|---|
| 役割 | キャラ設定（一人称・複数形・口調・呼称）の取り違えを検出する |
| 検出ポイント | クロちゃんが「私」「私たち」を使う / 幸田姉さんが「ぼく」「ぼくら」を使う / 幸田姉さんがクロちゃんを「あんた」以外で呼ぶ / 記事冒頭以外で「幸田姉さん」フル表記が使われる / 名前由来やキャラ設定が会話内に混入する / 社長が話者として対話に登場する |
| 参照ファイル | `.claude/skills/write-hatena-diary/quality-guidelines.md` Part 1「登場人物」「2 人の関係」「オーナー（社長）の位置づけ」 |
| プロンプト制約 | ガイドライン準拠・言及正確性・対話整合性の指摘は出さない（他観点エージェントが担当する）。Part 1（物語世界）の遵守チェックに専念する |

### 2. ガイドライン準拠

| 項目 | 内容 |
|---|---|
| 役割 | `quality-guidelines.md` Part 2「品質ルール」の各節と記事本文を機械的に照合し、違反を検出する |
| 検出ポイント | タイトル方針違反 / 1 H2 1 話題違反 / 内部実装語の初出説明セット欠落 / 地の文の配置違反（H2 セクション頭のシーン説明以外に会話以外の文章を書かない）/ 簡素記法の不備（マーカー残存・閉じ忘れ・入れ子）/ リポジトリ参照ルール違反（`owner/name` 形式 / GitHub URL / Issue 番号・PR 番号の本文混入）/ 個人・環境固有情報の混入 / 著作物・IP 情報の混入（実在作品由来の決め台詞・固有口調等の暗黙引用。ただし `:::bluesky` 引用ブロック内および引用を受ける文脈の本文での言及は許容） |
| ソフト推奨の扱い | 「1 セリフ 1 論点を推奨」「同一話者の連続発話は原則避け」等のソフト推奨はステップ 5「書き手スキップ」基準に該当するため、観点 2 では指摘対象外 |
| 参照ファイル | `.claude/skills/write-hatena-diary/quality-guidelines.md` Part 2 全節 + `.claude/skills/write-hatena-diary/template-diary.md` + `.claude/skills/write-hatena-diary/balloon-html.md` + `~/.claude/rules/invariants.md`「著作物・IP 情報の混入防止」 |
| プロンプト制約 | キャラ整合性・言及正確性・対話整合性の指摘は出さない。Part 2 機械照合と著作物・IP 検出に専念する |

### 3. 言及正確性

| 項目 | 内容 |
|---|---|
| 役割 | 記事本文の事実関係（リポ名・スキル名・固有名詞・ジャーナルから派生した出来事の描写）が素材ジャーナルと整合しているかを検出する |
| 検出ポイント | 存在しないリポ名・スキル名の言及 / リポ名のタイポ / ジャーナルに記載のない出来事を事実として書く / 推測・憶測を事実として書く / 数値・日付・固有名詞の誤り / Bluesky 投稿の `created-at`（JST）と本文での時間帯描写・前後対話文脈の整合性（明示的な「朝/昼/夕方/夜」表現に限らず、暗示的な時系列描写も含む） |
| 参照ファイル | 対象記事のフロントマター `date:` の日付に紐づくジャーナル本文（処理手順 2 で取得・サブエージェントへ渡す） + リポジトリマスターテーブル（`.claude/skills/write-hatena-diary/template-diary.md`） + 対象記事内の `{{{bluesky ... }}}` ブロックの `created-at`（JST） |
| プロンプト制約 | キャラ整合性・ガイドライン準拠・対話整合性の指摘は出さない。素材ジャーナルとの照合による事実誤認の検出に専念する |

### 4. 対話整合性

| 項目 | 内容 |
|---|---|
| 役割 | 姉さん・クロちゃんの問いと返しの主語・対象・経路の噛み合いを検証する |
| 検出ポイント | 姉さんの問いとクロちゃんの返しで主語が一致しない / 推測ベースの問いに対して能動経験ベースで答える等の論理経路ずれ / ジャーナル原文の出来事経路（誰が誰に指示したか・誰が確認に動いたか・能動受動関係）と記事の対話描写の不一致 / 直前のセリフを受けていない飛躍した返し |
| 参照ファイル | 対象記事のフロントマター `date:` の日付に紐づくジャーナル本文（処理手順 2 で取得・サブエージェントへ渡す） |
| プロンプト制約 | キャラ整合性・ガイドライン準拠・言及正確性の指摘は出さない。問いと返しの噛み合い、および対話に登場する主語・経路がジャーナル原文と整合するかに専念する |

### 観点設計の意図

観点は日記特性に必要なものに絞る。物語性の高い日記に対してレビュー観点が多すぎると、強制力が強くなり面白さが落ちるリスクがあるため、「明らかな間違い・禁止事項・発言ミス」に集中する設計判断。

## 処理手順

### 1. 引数バリデーション

`$ARGUMENTS` を空白で分割し、`--auto-publish` フラグの有無を判定する。残りのトークンのうち最初の非空トークンを対象ファイルパス `TARGET_FILE` とする。

- `--auto-publish` トークンが含まれる場合は `AUTO_PUBLISH=1` とし、`TARGET_FILE` の決定からは除外する。`--auto-publish` 以外の `--` 始まりトークンを受け取った場合は `エラー: 未知のオプションです: <トークン>` で停止する
- 第 1 トークンが空の場合: エラー停止: `エラー: 対象ファイルパスを指定してください`
- 絶対パス（以下のいずれかにマッチする場合）はエラー停止: `エラー: 対象ファイルパスはリポジトリルートからの相対パスで指定してください: <TARGET_FILE>`
  - 正規表現 `^[A-Za-z]:[/\\]`（Windows ドライブレター付き絶対パス、例: `C:/foo`、`C:\foo`）
  - 正規表現 `^/`（Unix 絶対パス、例: `/home/foo`）
  - 正規表現 `^\\\\`（Windows UNC パス、例: `\\server\share`）
- `Bash` ツールで `[ -f "$TARGET_FILE" ]` を実行してファイル存在を確認する。存在しない場合: エラー停止: `エラー: 対象ファイルが見つかりません: <TARGET_FILE>`
- 拡張子が `.md` 以外の場合: エラー停止: `エラー: 対象ファイルは Markdown (.md) を指定してください: <TARGET_FILE>`
- パスが `articles/hatena/` 配下でない場合: エラー停止: `エラー: 本スキルは articles/hatena/ 配下の日記記事を対象としています: <TARGET_FILE>`

### 2. 素材取得（ジャーナル参照観点向け）

ジャーナル本文を参照する観点（言及正確性・対話整合性）のサブエージェントに渡す素材を取得する。

なお、後続のサブエージェントプロンプトには `TARGET_FILE` の絶対パスを埋め込む必要があるため、本ステップ冒頭で `TARGET_FILE_ABS = "$(pwd)/$TARGET_FILE"` を構築しておく（リポジトリルートで実行されている前提）。

1. `Read` で `TARGET_FILE` を読み、フロントマター `date:` から記事の日記対象日 `ARTICLE_DATE` を抽出する
2. フロントマター `date:` が存在しない・形式が不正な場合: 警告を表示し、ジャーナル参照観点は **ジャーナル照合を行わないモード**（言及正確性はリポジトリマスターテーブルのみ参照、対話整合性は対話の主語経路の論理整合性のみチェック）で起動する。キャラ整合性・ガイドライン準拠は通常通り起動
3. `rag_list_by_date_range(date_from=ARTICLE_DATE, date_to=ARTICLE_DATE, source_type="journal", limit=100)` で当日のジャーナル一覧を取得
4. 各ジャーナルについて `rag_get_document(source_id=...)` で全文取得し、`JOURNAL_TEXTS` として保持する。個別取得失敗時は警告を表示し該当ジャーナルをスキップ、取得済みのみで実行する（全件失敗時はステップ 5 と同じくジャーナル照合なしモードに降格）
5. ジャーナルが 0 件の場合: 警告を表示し、ジャーナル参照観点はジャーナル照合なしモードで起動する

### 3. サブエージェント並列起動

各観点について `Task` ツールで `subagent_type=general-purpose` を **並列起動** する。1 メッセージ内で観点数分の Task tool use を含めることで並列実行する。

各エージェントへのプロンプトは以下の構造で構築する:

```text
あなたははてなブログ向け日記記事を「<観点名>」観点のみでレビューするエージェントです。

## 対象ファイル
<TARGET_FILE_ABS（ステップ 2 で構築した絶対パス）>

## あなたの担当観点
<観点 N の役割・検出ポイント・プロンプト制約をそのままコピー>

## 参照ファイル
<観点 N の参照ファイル一覧（絶対パス）>

## 素材ジャーナル（言及正確性・対話整合性のみ）
<JOURNAL_TEXTS を埋め込む。事実関係照合（言及正確性）と対話の主語経路照合（対話整合性）に使用する>

## 制約
- 担当観点の指摘のみ出してください。他観点（<他観点リスト>）には踏み込まないでください
- 指摘がない場合は「指摘なし」と返してください
- 推測や曖昧表現を避け、具体的な行番号・該当箇所を引用してください

## 出力フォーマット
以下の markdown セクション形式で返してください。**「提案: 」プレフィックスは厳守**（後続の triage 連携で文字列マッチングに利用するため、「修正案」「修正提案」等への言い換えは不可）:

### <観点名>

- **指摘 1**: <該当箇所（行番号 or 引用）>
  - 問題: <何が問題か>
  - 提案: <どう直すか（具体的な置換文字列・diff・修正方針を明示）>
- **指摘 2**: ...

(指摘がない場合は「指摘なし」のみ)

**`--auto-publish` 経路での追加制約**: 呼び出し元が `AUTO_PUBLISH=1` の場合、「提案」フィールドは以下の構造化形式で記述する（書き手自己判断ステップ 6.5 が `Edit` ツールで機械的に適用するため）。生テキストを 2 つの fenced code block に分けて配置し、エスケープを不要にする:

提案フォーマット例（記述例。実出力では `提案:` の下にこの 2 ブロックを置く）:

- 置換前ブロック: ` ```text ` で囲み、対象記事から抜粋した置換前文字列の生テキストをそのまま配置
- 置換後ブロック: ` ```text ` で囲み、置換後文字列の生テキストをそのまま配置

`old_string` は対象記事内で **一意に特定できる文字列** とし、前後の文脈を含めて 1〜3 行で抜粋する。複数行にわたる置換も同じ fenced code block 内に改行込みで配置する。具体的置換が定義できない指摘（口調調整等のソフト推奨）は提案を出さない（指摘自体を出さない、または「提案なし」と明示する）。
```

### 4. 結果集約 + ファイル出力

並列起動した 3 サブエージェントの結果を回収し、観点順に連結する。

出力先パスの組み立て:

- ディレクトリ: `.tmp/review-hatena-diary/`（存在しなければ `Bash` ツールで `mkdir -p` で作成）
- ファイル名: `<日時>-<slug>.md`
  - `<日時>`: `YYYYMMDD-HHMMSS` 形式（`Bash` ツールで `date '+%Y%m%d-%H%M%S'` で取得）
  - `<slug>`: 対象ファイル名から拡張子を除いた値（例: `articles/hatena/2026-05-16-11-46-00-foo.md` → `2026-05-16-11-46-00-foo`）

集約ファイルのフォーマット:

```markdown
# 日記レビュー結果

- 対象ファイル: `<TARGET_FILE>`
- 実行日時: <ISO8601>
- 参照ジャーナル件数: N 件（言及正確性・対話整合性用）

## サマリ

| 観点 | 指摘件数 |
|---|---|
| キャラ整合性 | N |
| ガイドライン準拠 | N |
| 言及正確性 | N |
| 対話整合性 | N |

## 詳細

<各観点エージェントの出力（### 見出し付きセクション）をそのまま連結>
```

サマリ表の件数列の値:

- サブエージェント実行が失敗した観点 → `エラー`（数値ではなく文字列で記録）。合計件数の計算からは除外
- サブエージェントが正常応答した観点 → 集約ファイル生成後、当該観点の `###` 見出しから次の `###` 見出しまでの範囲内で `- **指摘` 始まりの行を awk で範囲抽出してカウントする。観点をまたいだ誤集計を防ぐため、必ず当該観点の見出し範囲内に絞る:

  ```bash
  awk '/^### キャラ整合性$/{f=1;next} /^### /{f=0} f && /^- \*\*指摘/' <OUTPUT_FILE> | wc -l
  ```

### 5. 指摘の自動分類

集約ファイルから全指摘を抽出し、本スキル駆動 Claude が以下のクリティカル基準で各指摘を **クリティカル** または **書き手スキップ** に自動分類する。

#### クリティカル（triage でユーザー確認が必要）

- キャラ整合性違反（一人称・呼称の取り違え・社長が話者として登場等）
- 簡素記法の構文エラー（閉じ忘れ・入れ子・マーカー残存）
- 個人・環境固有情報の混入
- 著作物・IP 情報の混入（`quality-guidelines.md`「著作物・IP 情報の取り扱い」の例外条項を超える範囲）
- 事実誤認（言及正確性観点で素材ジャーナルと矛盾する記述）
- 対話経路の不一致（対話整合性観点でジャーナル原文と矛盾する主語・経路）

#### 書き手スキップ（triage 不要、自動で見送る）

日記は読み物として面白さ・リズムを優先する。以下はクリティカルでない限り書き手判断でスキップしてよい:

- 修正によって話の面白み・キャラの掛け合いのテンポが損なわれる指摘
- 1 セリフの長さ・論点数等、量的な目安に対する厳格適用
- 言い回し・語尾の好みレベルの調整提案
- 内部実装語の初出説明セット形式の機械チェック違反（読者が文脈で意味を取れる場合）

#### 分類結果の扱い

- `AUTO_PUBLISH=0`（通常モード）:
  - クリティカル指摘が 1 件以上 → ステップ 6 で `/triage` を起動
  - クリティカル指摘が 0 件 → triage をスキップしてステップ 8 へ進む
  - 書き手スキップ指摘はすべて自動でスキップ扱いとし、判断理由を内部記録する
- `AUTO_PUBLISH=1`（無人モード）:
  - **書き手スキップ指摘** はすべて自動でスキップ扱いとする（通常モードと同じ。日記の面白さ・テンポを優先するソフト推奨は無人実行でも採用しない設計）
  - **クリティカル指摘** のみステップ 6 をスキップし、ステップ 6.5 で書き手自己判断を行う

### 6. 指摘の triage 連携（`AUTO_PUBLISH=0` かつクリティカル指摘がある場合のみ）

`AUTO_PUBLISH=1` の場合は本ステップをスキップしてステップ 6.5 へ進む。

クリティカル指摘が 1 件以上ある場合、`/triage` スキルを `Skill` ツールで起動する。triage の分類・確認手段の選択・修正方針確定の動作仕様は `~/.claude/skills/triage/SKILL.md` を SSoT として参照する。

triage への引数には以下を含める:

- 各クリティカル指摘の対象箇所（ファイル名:行番号）
- 各指摘の要約
- 各指摘の「提案」（観点プロンプトの出力フォーマット「提案: 」フィールド）
- 全指摘の対象ファイルが `TARGET_FILE` であること

triage が中断（ユーザーが途中で打ち切り）された場合: それまでに確定した修正のみ次ステップで適用する。確定していない指摘はスキップ扱い。

### 6.5 書き手自己判断（`AUTO_PUBLISH=1` の場合のみ）

`AUTO_PUBLISH=0` の場合は本ステップをスキップしてステップ 7 へ進む。

無人実行のためユーザー確認が取れない代わりに、本スキル駆動 Claude が各クリティカル指摘を以下の機械的基準で 3 分類する:

| 分類 | 判定基準 | 後続処理 |
|---|---|---|
| **自動採用** | 「提案」フィールドの 2 つの fenced code block から置換前文字列 / 置換後文字列が抽出可能（ステップ 3 の出力フォーマット制約を満たす）。かつ `TARGET_FILE` 全文内に置換前文字列が **完全一致で 1 回のみ** 出現する | ステップ 7 で Edit 適用 |
| **棄却（前提誤り）** | 「提案」から置換前文字列は抽出できるが、`TARGET_FILE` 全文内に出現しない（書き手の架空指摘・引用ミス・別ファイルの指摘混入等） | スキップ。理由を内部記録 |
| **棄却（具体性不足）** | 「提案」から 2 つの fenced code block を抽出できない（自由作文・抽象的方針・「提案なし」）、または置換前文字列が `TARGET_FILE` 全文内に 2 回以上出現する（一意性なし） | スキップ。理由を内部記録（手動運用での再レビュー候補として残す） |

判定対象はクリティカル指摘のみ（書き手スキップ指摘はステップ 5 で既に「自動スキップ扱い」として確定済み）。

判定手順:

1. クリティカル指摘ごとに、「提案」フィールドの 2 つの ` ```text ` fenced code block から置換前文字列・置換後文字列を抽出する
2. 抽出に失敗（fence の数が 2 でない・順序不明）したら **棄却（具体性不足）** に分類
3. 抽出成功なら `Read` ツールで `TARGET_FILE` 全文を読み、Python 等で `text.count(old_string)` 相当（リテラル完全一致カウント。正規表現は使わない）を実施する。0 件 → **棄却（前提誤り）**、2 件以上 → **棄却（具体性不足）**、1 件 → **自動採用**

判定結果を内部リストに記録し、ステップ 7 で「自動採用」の指摘のみを Edit 適用する。

### 7. 修正適用

採用された指摘について、`Edit` ツールで `TARGET_FILE` に修正を適用する。採用判定の出所はモードで異なる:

- `AUTO_PUBLISH=0`（通常モード）: triage で「修正する」または「別案がある」と確定した指摘
  - 「修正する」: 観点エージェントの「提案」を採用
  - 「別案がある」: ユーザーが triage で提示した別案を採用
- `AUTO_PUBLISH=1`（無人モード）: ステップ 6.5 で「自動採用」と判定した指摘
  - 観点エージェントの「提案」をそのまま採用

修正適用は 1 指摘ずつ Edit を呼び出す（複数指摘を 1 Edit でまとめて適用しない）。以下の場合は **適用失敗として警告を表示しスキップ**、残りの指摘の修正を続行する:

- 観点エージェントの「提案」が `old_string` / `new_string` 形式で揃わない・整形不能（自由形式の修正方針のみで具体的な置換文字列が抽出できない場合等。本スキルでは無理に整形しない）
- `Edit` 実行時に `old_string` が一意でない・存在しない等の理由で失敗する

修正適用後の **再レビューは行わない**。1 回の review → triage → 修正で完結する。

### 8. チャット表示

集約ファイルは自動オープンしない。ユーザーは表示されたパスから手動で開く。チャットには **ファイルパス・観点別件数・自動分類結果・修正適用結果** を表示する。

表示フォーマットは `AUTO_PUBLISH` の値で出し分ける。

#### `AUTO_PUBLISH=0`（通常モード）

```text
✅ /review-hatena-diary 完了

📁 対象記事: <TARGET_FILE>
📁 レビュー結果: .tmp/review-hatena-diary/<TIMESTAMP>-<SLUG>.md

📊 観点別指摘件数:
  - キャラ整合性: N 件
  - ガイドライン準拠: N 件
  - 言及正確性: N 件
  - 対話整合性: N 件

🔀 自動分類:
  - クリティカル: N 件（triage へ）
  - 書き手スキップ: N 件

🛠️ 修正適用:
  - 適用: N 件
  - スキップ: N 件
  - 適用失敗: N 件

合計: N 件
```

#### `AUTO_PUBLISH=1`（無人モード）

```text
✅ /review-hatena-diary 完了（auto-publish）

📁 対象記事: <TARGET_FILE>
📁 レビュー結果: .tmp/review-hatena-diary/<TIMESTAMP>-<SLUG>.md

📊 観点別指摘件数:
  - キャラ整合性: N 件
  - ガイドライン準拠: N 件
  - 言及正確性: N 件
  - 対話整合性: N 件

🔀 書き手自己判断:
  - 自動採用: N 件
  - 棄却（前提誤り）: N 件
  - 棄却（具体性不足）: N 件

🛠️ 修正適用:
  - 適用: N 件
  - 適用失敗: N 件

合計: N 件
```

表示ルール（両モード共通）:

- **エラー観点がある場合**: 該当観点は `<観点名>: エラー（<理由要約>）` と表示し、合計件数の計算には含めない。合計表示の末尾に `（エラー: M 観点）` を付記する
- **クリティカルが 0 件の場合**（通常モードのみ）: `triage スキップ（クリティカル指摘なし）` と明示
- **修正適用失敗が 0 件の場合**: `適用失敗: 0 件` の行は省略してよい

## エラーハンドリング

| 状況 | 対応 |
|---|---|
| 引数なし | エラー表示して停止 |
| 未知のオプション（`--auto-publish` 以外の `--` 始まりトークン） | エラー表示して停止 |
| 絶対パス指定 | エラー表示して停止 |
| 対象ファイルが存在しない | エラー表示して停止 |
| 拡張子が `.md` でない | エラー表示して停止 |
| `articles/hatena/` 配下でない | エラー表示して停止 |
| フロントマター `date:` 不在・不正 | 警告表示 + ジャーナル参照観点（言及正確性・対話整合性）をジャーナル照合なしモードで起動 |
| `rag_list_by_date_range` 失敗 | 警告表示 + ジャーナル参照観点をジャーナル照合なしモードで起動（他観点は継続） |
| 当日ジャーナル 0 件 | 警告表示 + ジャーナル参照観点をジャーナル照合なしモードで起動 |
| サブエージェント実行失敗（一部） | 失敗した観点をサマリに `エラー` として記録し、他観点の結果は継続して集約・triage に進む |
| 全サブエージェント失敗 | エラー表示して停止（triage 連携・修正適用はスキップ） |
| `rag_get_document` 個別失敗（一部ジャーナル取得失敗） | 警告表示 + 該当ジャーナルをスキップ、取得済みのみで実行。全件失敗時はジャーナル照合なしモードに降格 |
| `.tmp/` ディレクトリ作成失敗 | エラー表示して停止 |
| クリティカル指摘 0 件 | `AUTO_PUBLISH=0` では triage をスキップしてチャット表示へ進む。`AUTO_PUBLISH=1` でもステップ 6.5（書き手自己判断）の対象がないためスキップしてチャット表示へ進む |
| triage 中断（ユーザー打ち切り） | それまでに確定した修正のみ適用、未確定はスキップ扱いで続行。`AUTO_PUBLISH=1` では triage 自体を実行しないため該当しない |
| `AUTO_PUBLISH=1` で「自動採用」が 0 件 | 警告は出さず修正適用 0 件のままチャット表示へ進む（指摘の前提誤りや判断保留が多いケース。手動運用で再確認する想定） |
| Edit による修正適用失敗（`old_string` 不一致等） | 警告表示 + 該当指摘をスキップして残りを続行。チャット表示の「適用失敗」に件数を計上 |
