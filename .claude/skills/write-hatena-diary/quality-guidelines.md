# 執筆品質ガイドライン (write-hatena-diary)

本ファイルははてなブログ向け日記記事の執筆品質ルール。`/write-hatena-diary` スキルが記事生成時に参照する
ほか、`/multi-perspective-review` のガイドライン準拠チェック観点が本ファイルを参照する。

## タイトル

- **内容を示す自然な日本語**（20〜30 文字目安）
- **日付は含めない**（日付はフロントマター・ファイル名で表現済み。タイトルに重複させない）
- その日のジャーナル内容から最も特徴的な作業・出来事を 1〜2 個取り上げて要約する
- NG: `2026-05-13 開発日記` のような日付固定タイトル（情報量がゼロに近い）
- 良い例:
  - `ジャーナル機能の追加とリポ呼称ルールの整備`
  - `テンプレート刷新でドキュメント体系を見直す`
  - `日記スキルの設計を進めた`

## 構成

- **セクション構成は `template-diary.md` の「記事本文セクション雛形」群を SSoT とする**（ジャーナルテンプレート準拠ハイブリッド方式）
- 本ファイルでは構成そのものを再掲しない（同期負担回避のため）
- 構成の遵守確認:
  - `## 今日の作業` 配下の `### 概要` が必須セクションとして存在するか
  - `## ペルソナの感想` `## 明日に向けて` `## プロジェクトの説明` が存在するか
  - 条件付きセクション（`### 判断` / `### うまくいった / ハマった` / `### 気づき` / `### 残課題` / `## オーナーへの一言` / `## オーナーの Bluesky 投稿への反応`）が必要な素材があるときのみ含まれているか

## 文字数

- 日記記事は **1,000〜3,000 字** を目安（技術記事より軽め、エンタメ要素を含むため）
- 短すぎる場合: ジャーナルの「判断」「気づき」を掘り下げる、ペルソナの感想を 1〜2 段落足す
- 長すぎる場合: 「今日の作業」の詳細を絞る、セクションを統合する

## 文体

- 文体に関するペルソナ仕様（一人称・オーナー呼称・トーン使い分け）は `persona.md` を SSoT として参照する

## 記法

- **GitHub-flavored Markdown（GFM）のみ**で記述する
- **はてな記法は使用しない**（`[id:foo:title]` 形式・`*` / `**` の独自意味付け・`>>>` 引用ブロック等）
- Markdown と競合する記号は意図に応じてエスケープする（文字としての `_` は `\_`）
- 詳細: `template-diary.md` の「記法ポリシー」セクションを参照

## リポジトリ参照

- 本文では `name` を backtick 付きで直接書く（例: `` `rag-knowledge` ``、`` `article-writer` ``）
- GitHub `owner/name` 形式・URL・Issue 番号・PR 番号は本文に書かない
- 「こういう Issue があって、こういう対応をした」レベルの抽象的言い回しは可
- リポの個別説明は記事内に書かない（外部記事に集約。URL は `template-diary.md` 「`## プロジェクトの説明` (末尾固定)」セクションを SSoT として参照）
- 記事末尾の `## プロジェクトの説明` セクションで外部記事へのリンクを固定文言で提示する

## 感想セクションの書き方

- 日記の「気持ち」を出す唯一のセクション。技術ジャーナル寄りトーンの作業説明と差別化する
- 1〜3 段落程度。長くしすぎない
- 絵文字の使用範囲は `persona.md` セクション 4 を SSoT として参照する（記述形式は `template-diary.md` の「記法ポリシー」を参照）
- ジャーナルに書かれていない情緒・気持ちを書く場所。作業内容の重複は避ける

## オーナーへの一言

- ジャーナル「ユーザーへの意見」セクションがあるときのみ採用
- ペルソナ視点で「オーナーへ伝えたい」形に翻訳する
- 内容なしなら本セクション全体を省略（空セクション「特になし」を書かない）

## Bluesky 投稿への反応

- AI / LLM / 開発関連と判定された投稿のみ引用
- 引用は **`scripts/generate_bluesky_embed.py` で生成した HTML 埋め込みスニペット** を Markdown 本文に挿入する（はてなブログ Markdown モードでプレビュー時点で投稿カードとして表示される）
- 1 投稿 = 1 スニペット。複数投稿があればスニペットとペルソナのコメントを繰り返す
- スニペット内に DID / handle / 投稿日時が含まれるため、本文中で別途これらを書き出す必要はない
- 該当投稿が 0 件なら本セクション全体を省略
- フォーマットの詳細は `template-diary.md` の「Bluesky 引用フォーマット」を参照

## 明日に向けて

- 残課題やこれから取り組みたいことを軽めのトーンで
- 残課題が特になければ「特になし」と書く（簡潔に）
- 翌日の予告として機能させる（読者の継続読み動機になる）

## 一次情報の重視

- 「やってみたらこうだった」を書く際は、事実関係を歪めない
- ジャーナルにない情報を持ち込まない（推測・憶測を事実として書かない）
- 未確認の内容は「ドキュメントによると」「一般的には」と注記する

## 内容の正確性

- **未検証の機能・環境・実行条件を断定的に書かない**
- **「最も」「必ず」「絶対」等の最上級・断定表現を避ける**
- **スコープを守る**: 日記の主題から逸れる情報は含めない（その日に扱っていない話題を持ち込まない）
- **コスト・料金情報は慎重に**: 個別契約に依存する情報は具体的な金額を避けるか前提条件を明示
- **記事内の整合性をチェック**: 異なるセクション間で前提・結論が食い違っていないか確認

## 個人・環境固有情報の混入禁止

`.claude/skills/write-zenn/quality-guidelines.md`「個人・環境固有情報の混入禁止」と同じ規則を適用する。

書いてはいけないもの:

- 絶対ローカルパス（`C:/Users/...`、`D:/GitHub/...`、`/Users/<name>/...`、`/home/<name>/...` 等）
- 実セッション ID、UUID、トークン、ハッシュ等の実値
- 認証情報の値（API キー・パスワード等）
- private リポへの URL リンク（リポ URL は本文・末尾セクションのいずれにも書かない。外部記事側で管理）
- 実在の人物・作品由来の固有名詞、台詞、決め台詞
- GitHub アカウント名（本文中での `becky3` 等の個人識別子）

## 著作物・IP 情報の混入防止

`~/.claude/rules/invariants.md`「著作物・IP 情報の混入防止」に準じる。

- 他者の Bluesky 投稿の引用は行わない（rag-knowledge 側にはオーナーの投稿のみ取り込まれている前提）

## ペルソナ調整に関する自己制御

- 記事生成中にペルソナの調整が必要と気づいた場合、**勝手に書き換えず、オーナーに確認**してから変更する
- 単一記事内でその場限りのトーン調整（例: ある日の感想セクションを少し抑え目にする）は許容
- 「次回以降ずっとそうする」種類の調整はオーナー確認が必要
- 確認なしで変更できるのは typo 修正のみ

## レビュー観点との対応（multi-perspective-review）

`/multi-perspective-review` から本ファイルを参照する観点は「ガイドライン準拠チェック」。
本ファイルの各節と記事本文を機械的に照合し、違反があれば指摘する。
