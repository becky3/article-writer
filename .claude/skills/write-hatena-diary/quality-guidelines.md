# 執筆品質ガイドライン (write-hatena-diary)

本ファイルははてなブログ向け日記記事の執筆品質ルール。`/write-hatena-diary` スキルが記事生成時に参照する
ほか、`/multi-perspective-review` のガイドライン準拠チェック観点が本ファイルを参照する。

## 記事の形式と人格

- 2 人のキャラクター（クロちゃん・幸田姉さん）の対話形式で記述する
- キャラクターの動機・口癖・話し方・関係性は `persona.md` を SSoT として参照する
- 本ファイルでは人格仕様を再掲しない

## タイトル

- **内容を示す自然な日本語**（20〜30 文字目安）
- **日付は含めない**（日付はフロントマター・ファイル名で表現済み）
- その日のジャーナル内容から最も特徴的な作業・出来事を 1〜2 個取り上げて要約する
- NG: `2026-05-13 開発日記` のような日付固定タイトル（情報量がゼロに近い）

## 構成

- **本文の構成は `persona.md` を SSoT** とする（書き手の裁量で対話シーンを組み立てる）
- 機械的に守るべき固定要素は `template-diary.md` 「記事の機械的出版要件」を参照
  - フロントマター (title / date / category)
  - リポジトリ言及時の `name` backtick / `<code>` 表記
  - プロジェクト説明ページへの固定リンク（冒頭または末尾、配置は記事側で柔軟）
- 各話題ごとに端的な H2 見出しを置き、短い地の文 + 吹き出し対話で展開する
- 旧仕様の固定セクション（`## 今日の作業` `## ペルソナの感想` `## オーナーへの一言` `## オーナーの Bluesky 投稿への反応` `## 明日に向けて` 等）は廃止

## 文字数

- 日記記事は **1,500〜3,000 字** を目安
- 短すぎる場合: 対話のシーンを増やすか、ジャーナルから別ネタを拾う
- 長すぎる場合: シーン数を絞る、または短いネタを統合する

## 文体

- 文体仕様（一人称・二人称・口調・口癖・呼称ルール・話し方のクセ）はすべて `persona.md` を SSoT として参照する
- セリフは生成時に書き手が自分で組み立てる（persona.md の 6 要素から推論する）

## 記法

- **GitHub-flavored Markdown（GFM）のみ** で記述（はてな記法は使用しない）
- **HTML タグ内では Markdown 構文が解釈されない**。リポ名・スキル名等は `<code>` タグで囲む（詳細は `template-diary.md` 「記法ポリシー」参照）
- 吹き出し・キャラ紹介カードの class 名・構造は `hatena-design.css` を SSoT として参照
- 記事内に `<style>` を埋め込まない（CSS ははてなブログのデザイン CSS で管理済み前提）

## リポジトリ参照

- 本文では `name` を backtick（通常文中）または `<code>` タグ（HTML タグ内）で直接書く
- GitHub `owner/name` 形式・URL・Issue 番号・PR 番号は本文に書かない
- 抽象的な言い回し（「あの Issue で〜」「先週のリリース」等）は可
- リポの個別説明は記事内に書かない（外部記事に集約）
- プロジェクト説明ページへの固定リンクは `template-diary.md` 「末尾固定セクション」を参照

## オーナー（社長）の扱い

- 社長は直接登場しない。2 人の会話の中で言及される（詳細は `persona.md` セクション 3 参照）
- 本気の批判・揶揄はしない。愛のあるからかい範囲を守る
- 社長本人が読んでも気持ちがいい範囲を守る（共感を狙うぼやき、毒舌は避ける）

## Bluesky 投稿の扱い

- 引用は `scripts/generate_bluesky_embed.py` で生成した HTML 埋め込みスニペット必須
- 1 投稿 = 1 スニペット
- 引用の演出は `persona.md` セクション 3 「社長の SNS（Bluesky）について」を SSoT として参照
  - 「上司の素顔をのぞき見できる秘密のチャンネル」を 2 人だけが見つけて秘密共有する演出
  - 暴露的・愛のあるからかいのトーン
- 該当投稿が 0 件なら Bluesky 関連シーンを省略
- 他者の投稿（オーナー以外）は引用しない

## 一次情報の重視

- 「やってみたらこうだった」を書く際は事実関係を歪めない
- ジャーナルにない情報を持ち込まない（推測・憶測を事実として書かない）
- 未確認の内容は「ドキュメントによると」「一般的には」と注記する

## 内容の正確性

- 未検証の機能・環境・実行条件を断定的に書かない
- 「最も」「必ず」「絶対」等の最上級・断定表現を避ける
- スコープを守る: 日記の主題から逸れる情報は含めない
- コスト・料金情報は慎重に: 個別契約に依存する情報は金額を避けるか前提条件を明示
- 記事内の整合性: 異なるシーン間で前提・結論が食い違わないか確認

## 個人・環境固有情報の混入禁止

`.claude/skills/write-zenn/quality-guidelines.md`「個人・環境固有情報の混入禁止」と同じ規則を適用する。

書いてはいけないもの:

- 絶対ローカルパス（`C:/Users/...`、`D:/GitHub/...`、`/Users/<name>/...`、`/home/<name>/...` 等）
- 実セッション ID、UUID、トークン、ハッシュ等の実値
- 認証情報の値（API キー・パスワード等）
- private リポへの URL リンク（リポ URL は本文・末尾セクションのいずれにも書かない。外部記事側で管理）
- 実在の人物・作品由来の固有名詞、台詞、決め台詞
- GitHub アカウント名（本文中での `becky3` 等の個人識別子）

## 著作物・IP 情報の混入防止

`~/.claude/rules/invariants.md`「著作物・IP 情報の混入防止」に準じる。

- キャラ名「クロちゃん」「幸田姉さん」は Claude / Claude Code 由来の音韻語呂であり、元作品の固有名詞ではない
- 本文中で実在作品名・元作品由来の決め台詞・固有口調を使用しない
- 他者の Bluesky 投稿の引用は行わない（rag-knowledge 側にはオーナーの投稿のみ取り込まれている前提）

## ペルソナ調整に関する自己制御

- 記事生成中にペルソナの調整が必要と気づいた場合、**勝手に書き換えず、オーナーに確認**してから変更する
- 単一記事内でその場限りのトーン調整は許容（例: あるシーンで姉さんの口数を少し抑える）
- 「次回以降ずっとそうする」種類の調整はオーナー確認が必要
- 確認なしで変更できるのは typo 修正のみ

## レビュー観点との対応（multi-perspective-review）

`/multi-perspective-review` から本ファイルを参照する観点は「ガイドライン準拠チェック」。
本ファイルの各節と記事本文を機械的に照合し、違反があれば指摘する。
