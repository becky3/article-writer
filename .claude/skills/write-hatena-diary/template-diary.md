# 日記テンプレート (write-hatena-diary)

本ファイルは `/write-hatena-diary` スキルが記事生成時に参照するテンプレート。
生成記事の本文と末尾「プロジェクトの説明」セクションを組み立てる。

## 記法ポリシー

本スキルは **GitHub-flavored Markdown（GFM）のみ** で記事を生成する。はてな記法（`[id:foo:title]` 形式、`>>>` 引用、`*` や `**` の独自意味付け等）は使用しない。Markdown と競合する記号（文字としての `_` など）は `\_` のようにバックスラッシュエスケープする。

**絵文字は Unicode 直接記述** を使う（例: ✨ 🙏 📝 ✍️ 🌱 🎉）。GFM ショートコード形式（`:sparkles:` `:pray:` `:memo:` 等）は **使わない**。はてなブログ Markdown モードはショートコードをレンダリングせず、そのまま `:sparkles:` という文字列として記事に表示されてしまう。

運用前提として、はてなブログ側のデフォルト編集モードを「Markdown」に設定しておく必要がある（「設定 → 基本設定 → 編集モード → Markdown」）。記事作成後に編集モードを変更できないため、本前提を満たさないと意図したフォーマットで表示されない。本前提はユーザー手動運用の範囲とし、スキルでは検知しない。

なお、**生成記事 (`articles/hatena/**`) はリポジトリの markdownlint 対象外** としている（フロントマター + H1 重複 / HTML 埋め込み / 日本語段落の line-length 超過が頻発し、機械的 lint の便益が低いため）。

## 記事フロントマター雛形

```yaml
---
title: "{記事タイトル}"
date: "{YYYY-MM-DD}"
category: "diary"
---
```

`title` / `date` / `category` の 3 項目のみ。`/publish-hatena` での下書き登録時に、これらの値が以下のように反映される:

- `title` → AtomPub Entry の `<title>`
- `date` → AtomPub Entry の `<updated>`（JST 0 時として ISO 8601 化、はてなブログ管理画面の公開予定日として表示）および `published.txt` 記録用の日付キー
- `category` → AtomPub Entry の `<category term="...">`

## 記事本文セクション雛形

### タイトル

```markdown
# {記事タイトル}
```

タイトルは **当該日のジャーナル内容を要約した自然な日本語** とする（20〜30 文字目安）。**日付は含めない**（日付はフロントマターとファイル名で表現される）。

例:

- 「ジャーナル機能の追加とリポ呼称ルールの整備」
- 「テンプレート刷新でドキュメント体系を見直す」
- 「日記スキルの設計を進めた」

その日の最も特徴的な作業・出来事を 1〜2 個取り上げて要約する。複数リポにまたがる作業を 1 文でまとめる必要はない（主要トピック中心で問題ない）。

セクション構成は <<### ステップ6: journal/ に新規ファイル作成@~/.claude/skills/handoff/SKILL.md>> をジャーナルテンプレート SSoT として、これと対応させたハイブリッド方式。

トップレベル `##` セクションのうち、`## 今日の作業` `## ペルソナの感想` `## 明日に向けて` `## プロジェクトの説明` の 4 つは必須。`## オーナーへの一言` `## オーナーの Bluesky 投稿への反応` は条件付き（該当素材があるときのみ）。

### 必須: ## 今日の作業 (技術ジャーナル寄りトーン)

ジャーナルの各項目を `###` サブセクションで反映する。サブセクションのうち `### 概要` のみ必須、他はジャーナルに該当記述があるときのみ採用。

```markdown
## 今日の作業

### 概要

{ジャーナル「概要」の要約 + 「結果」（良かった/悪かった/未検証）の統合}

### 判断

{ジャーナル「判断」: 迷った点・選んだ選択肢・理由}

### うまくいった / ハマった

{ジャーナル「うまくいったこと」「改善点・ハマったこと」を統合}

### 気づき

{ジャーナル「気づき・次に活かすこと」}

### 残課題

{ジャーナル「残課題」}
```

`## 今日の作業` 配下では複数リポにまたがる作業を扱う場合がある。
**リポは `name` を backtick 付きで本文に直接書く**（例: `` `rag-knowledge` ``、`` `article-writer` ``）。
読者にとって最初は意味が分からないリポ名があっても、末尾「`## プロジェクトの説明`」セクションの外部記事リンクから対応関係を辿れる。
`becky3/<name>` 形式・Issue / PR 番号は書かない。

### 必須: ## ペルソナの感想 (エンタメ寄りトーン)

```markdown
## ペルソナの感想

{ペルソナ目線のナラティブな振り返り。ジャーナルに書かれていない情緒・気持ちを書く場所}
```

絵文字は適度に使ってよい。1〜3 段落程度。

### 条件付き: ## オーナーへの一言 (エンタメ寄りトーン)

ジャーナル「ユーザーへの意見」セクションに該当記述がある場合のみ採用する。

```markdown
## オーナーへの一言

{ジャーナル「ユーザーへの意見」をペルソナ視点に翻訳した内容}
```

該当記述がなければ本セクション全体を省略する（空セクションを書かない）。

### 条件付き: ## オーナーの Bluesky 投稿への反応 (エンタメ寄りトーン)

Phase 3 の選別後、当該日に関連する Bluesky 投稿が 1 件以上残ったときのみ採用する。

```markdown
## オーナーの Bluesky 投稿への反応

{Bluesky HTML 埋め込みスニペット — 後述「Bluesky 引用フォーマット」セクションを参照}

{ペルソナのコメント}
```

関連投稿が 0 件の場合は本セクション全体を省略する（空セクションを書かない）。スニペット生成手順は後述「Bluesky 引用フォーマット」を参照する。

### 必須: ## 明日に向けて (軽めトーン)

```markdown
## 明日に向けて

{残課題を踏まえつつ、次にやりたいこと・気になっていること}
```

残課題が特になければ「特になし」と書く。

### 必須: ## プロジェクトの説明 (末尾固定)

リポジトリの個別説明は外部記事に集約する。記事末尾の最終セクションとして以下を固定で配置する:

```markdown
## プロジェクトの説明

リポジトリ名などの説明は以下を参照ください。

[プロジェクト説明](https://beckyjpn.hatenablog.com/entry/project)
```

- 本セクションは Phase 4 で言及リポの有無に関わらず常に挿入する（固定リンク）
- リポごとの実態説明・visibility 表記・GitHub URL は外部記事側で管理する

## Bluesky 引用フォーマット

Bluesky 投稿は、はてなブログ Markdown モードでもプレビュー時点で投稿カードとして表示されるよう、**`scripts/generate_bluesky_embed.py` で生成した HTML 埋め込みスニペット** を Markdown 本文に直接挿入する。

スニペットの生成は SKILL.md Phase 4 で行う（投稿ごとに以下のコマンドを実行し、stdout の HTML を取得）:

```bash
python scripts/generate_bluesky_embed.py \
  --did <DID> \
  --cid <CID> \
  --rkey <rkey> \
  --handle <handle> \
  --display-name <表示名> \
  --text "<投稿本文>" \
  --created-at <ISO 8601> \
  [--lang ja]
```

各投稿の HTML スニペットを記事本文に挿入し、その直後の段落にペルソナのコメントを書く:

```markdown
{HTML スニペット（複数行）}

{ペルソナのコメント。エンタメ寄りトーン。1〜2 段落}
```

引用は 1 投稿 = 1 スニペット。複数投稿があればスニペットとコメントを繰り返す。スニペット内に DID / handle / 投稿日時が含まれるため、本文中で別途これらを書き出す必要はない。
