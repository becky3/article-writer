# 執筆品質ガイドライン

記事生成時に必ず以下を適用する。Zenn/Qiita のトレンド調査から導出した指針。

## タイトル

- 「痛み」か「意外性」を含める（トレンド記事の共通特徴）
- 40-60文字を目安
- 効果的なパターン:
  - `{X}したら{Y}だった話` — 体験ベース
  - `{X}で{Y}を自動化した` — 成果ベース
  - `{X}の{N}つのポイント` — リスト系
  - `なぜ{X}で{Y}なのか？` — 疑問・問題提起系
  - `私が実際に使っている{X}だけの{Y}` — 限定・選別系
- NG: 抽象的すぎるタイトル（「クロスプラットフォーム対応について」等）

## 構成の原則

- **1記事1テーマ**: テーマがブレたら記事を分割する。詰め込みすぎない
- **結論ファースト**: 記事全体でもセクション内でも、結論・要点を先に書く。詳細は後に展開
- **前提の明示**: 環境・バージョン、前提知識を導入部で明確にする
- **流し読み対応**: 見出しだけで記事の大意が伝わる構造にする。太字で各段落の要点を強調
  - `###` セクション内に太字見出し（`**〜**`）が 3 つ以上あるなら `####` 小見出しに昇格させる

## 文字数

- 目標: 2,000〜6,000字（Zenn で Like が集まりやすいレンジ）
- 短すぎる場合: 具体例、代替アプローチ、背景の深掘りで補強
- 長すぎる場合: セクションの分割やシリーズ化を提案
- 注: 文字数より内容の質が重要。無理に引き延ばさず、簡潔さを優先する

## 文体

- です/ます調 + 適度な口語（堅すぎない、砕けすぎない）
- 開発者本人の体験として書く（一人称視点）。ただし一人称の使い方は **判断・選択の表現**（「〜と判断した」「〜を選んだ」）に寄せる
- `:::message` で重要な注意点を強調
- 不要なヘッジ表現（「かもしれません」「と思われます」）を避ける
- **物語調に流れやすい記述は、判断・結果として整理する**: 開発過程の経緯・気付き・試行錯誤は、語り口（時系列の物語）ではなく仕様・解決策の形に整理して書く。事例や試行錯誤の内容は一次情報として価値があるが、書き方は次のように調整する:
  - 「〜してみた」「最初は〜次に〜」型の時系列描写 → 「〜を試した結果、〜と判断した」のような判断・結果の形に
  - 「やってみた感想」「Before / After」型セクション → 技術的な選択と効果を整理した節に
  - 「執筆過程で〜が見つかった」型の経緯紹介 → 結果・洞察として記述
- **著者の試行錯誤・デバッグ過程を読者向け手順節に混ぜない**: 記事の主題が「経緯共有」でない限り、著者がどう失敗したか・どう発見したかという過程を、読者向け解説節（「こう読む／こう手順を踏む」系）に紛れ込ませない。経緯を書きたい場合は専用の事例セクションに集約する
- **節見出しと内容が重複する前置きを書かない**: 節の役割が見出しで明示されている場所で、その役割を改めて宣言する文（節導入の宣言文・直後に表や箇条書きが続く場面での前置き）は省く
- **読者視点で取捨する**: 自明な注記（「前提が満たされなければ機能しない」「依存先が変わると影響を受ける」等）、誤解の予防、経緯の説明、前提の補足は、読者がその情報を実際に必要とする場合のみ残す。記事の主張に直接寄与しないものは省く
- **記事執筆時点で事実でない経緯・要件は書かない**: 事後的に整えた架空の動機・要件提示で記事を構成しない。実際の動機をそのまま書くか、現在形で「本記事で扱う範囲」として提示する
- **記事の主題と関係ないツール名・実装名は参照しない**: ツール名・フィールド名・関数名（backtick 付き）は、そのツールの説明が必要な場合や読者が grep する対象になる場合に限り使う。それ以外の箇所では「AI が編集した」「ツールを呼び出した」のように一般語で表現する
- **括弧書きを多用しない**: 括弧書きは読みづらくなる主因のひとつ。用語注（初出時の英語／略語の補足）など必要な場合以外は使わず、本文に統合するか独立文に分離する

## 一次情報の重視

「やってみたらこうだった」を書く際の **内容** に関する指針。書き方（記述スタイル）については「文体」節の物語調禁止に従う（内容＝一次情報として歓迎、書き方＝物語ではなく仕様・解決策として整理）。

- 公式ドキュメントの焼き直しではなく「やってみたらこうだった」を書く
- エラー解決の試行錯誤プロセス、技術導入の成功・失敗事例に価値がある
- **実績・結果には具体例を添える**: 「12件処理」ではなく「docstring 追加やテスト補強といった定型タスク 12 件」のように、読者が自分のプロジェクトに当てはめられる粒度で書く
- 未確認の内容は「ドキュメントによると」「一般的には」と正直に注記する

## 主題（thesis）の扱い

- **ユーザー指示にない記事の主題（thesis）を勝手に作らない**: 題材を素直に紹介するスタンスを崩さない。差別化フックや thesis 的な見出しを記事に入れるかは、ユーザーの意図確認後に決める
- 本節を SSoT とし、`repo-scanner.md` および `SKILL.md` からは参照のみ行う

## 内容の正確性

- **未検証の機能を書かない**: 仕様書に定義されていても、実際に動作確認していない機能は記事に含めない
- **未検証の環境・OS・実行条件への言及禁止**: 動作確認していない環境について「同じ手順で動く」「対応しているはず」「他環境でも同様に動く」等の推測表現を書かない
- **安全性・動作の表現は正確に**: 「即座に停止」「必ず検出される」等の断定的な安全表現は、実際の挙動と一致しているか厳密に確認する（例: 「次のステップから停止」と「即座に停止」は異なる）
- **「最も」「必ず」「絶対」等の最上級・断定表現を避ける**: 比較対象を網羅的に検証していない限り使わない。「〜のほうが安定する」「〜と判断した」のように、自分の経験に基づく表現にする
- **実際に起きないシナリオを書かない**: 「Xが失敗したらYで停止」のような記述は、Xが実際に失敗するパスが存在するか確認する
- **スコープを守る**: 記事のテーマに直接関係しない情報は含めない。「あると便利かも」レベルでは入れない
- **コスト・料金情報は慎重に**: ユーザーの契約状況で変動する情報は、具体的な金額を避けるか、前提条件を明示する
- **用語は初出で定義する**: 専門用語・略語・英語の技術用語、および記事内で独自に作った造語は、初出箇所で日本語の補足を括弧書きで入れる（例: `Fine-grained PAT（リポジトリ単位で権限を絞れる個人アクセストークン）`）
- **技術的主張は裏取りしてから書く**: 「〜だけで動く」「〜は発生しない」等の技術的主張は、公式ドキュメント・Issue・ジャーナル・実際のコードで事実確認する。言葉の辻褄合わせで書かない
- **記事内の整合性をチェックする**: 記事の異なる箇所で書いた前提・結論が食い違っていないか確認する

## 個人・環境固有情報の混入禁止

記事に著者の PC・環境・プロジェクトに固有の情報を含めない。確認の余地なく禁止する。

- 絶対ローカルパス（`C:/Users/...`、`D:/GitHub/...`、`/Users/<name>/...`、`/home/<name>/...` 等）
- 実セッション ID、UUID、トークン、ハッシュ等の実値
- private リポジトリへのリンク（`.claude/sources.yml` に列挙されていてもリポが private なら参照しない）
- プロジェクト固有のスキル名・サブエージェント名・内部ルール名（汎用化または削除する。「自前のレビュー用エージェント」のような著者環境固有の概念も避け、「AI レビュー」のような一般語にする）

これらは本文・コードブロック・mermaid 図・参考リンク・画像プレースホルダーのいずれにも含めない。

## 導入の書き方

- **推奨: 事実提示型** — テーマと記事で扱う範囲を簡潔に示す
- **避ける**: 自己紹介から始める導入（フォロワーベースがない場合は逆効果）

## まとめの書き方

- 箇条書きで要点を整理
- 展望や応用可能性を1-2文で添える
- 「読んでくれてありがとう」等の定型文は不要

## 図表・コード

- **mermaid**: アーキテクチャ図、フローチャート、シーケンス図
  - 日本語ノードラベルは 1 行 6〜8 文字を目安に `<br/>` で改行（最大 3 行まで）。長文 1 行はノード幅でレンダリング時に見切れるため避ける
  - `~~~` 不可視リンクは「層境界の縦並べ」用途に限定する。フロー本筋のノード接続には使わない（斜め線が出る原因になる）
  - 戻り経路で実装上「処理なしで素通し」のケースは、図に含めず本文の注記で説明する
- **テーブル**: 比較、条件一覧、設計判断の根拠
  - 表の項目数が 2 行程度しかない場合は、表化せず説明文で書くことも検討する
  - テーブル内の情報が本文と重複する場合、テーブルは簡潔にして「（X 節参照）」で本文を参照させる
  - 列名・セル内容は読者に伝わる平易な表現にする（技術的な内部名だけでは不十分。例: `statusCheckRollup` → 「CI が全て成功しているか」）
- **コードブロック**: ファイル名注釈付き

  ````markdown
  ```python:process_guard.py
  ```
  ````

- **コードブロックの文脈設計（必須）**: 単にコードを貼らない。直前で「コードのどの観点に注目してほしいか」を 1 文示し、読者がコードを概念モデルへ橋渡しできる状態にする。「最小実装は次の通り」のような無情報な前置きだけでは不可

## 画像・スクリーンショット

- **画像はエンゲージメントに直結する**: Zenn の人気記事（いいね100超）は 4 枚以上の画像を使用している傾向がある。特に GitHub UI のスクリーンショットが効果的
- **プレースホルダー形式**: blockquote を使用（Zenn プレビューで視認できる）

  ```markdown
  > 📸 **TODO** — 画像の説明
  ```

- **`{{...}}` 形式は使わない**: プレビューで目立たず、公開時に差し替え忘れるリスクがある
- **1 プレースホルダ = 1 画像**: 複数の画像をまとめて書かない。テキストの流れの中に 1 枚ずつ配置する
- 概念図は mermaid で作成し、スクリーンショットは「動いている様子」「実際の UI 画面」に使う
- 再現可能なものだけ配置する（過去の状態で再現不能なスクショは入れない）
- **直前にリード文を必ず添える**: 「実際の動作はこんな感じです。」「次のような画面になります。」のように本文と画像をつなぐ 1 文を入れる。文脈に繋がらない場所には配置しない（唐突な貼り付けを避ける）
- **公式ドキュメントで参照可能なものは省略を検討**: 設定画面のスクショ等、公式ドキュメントへのリンクで補完できる場合は記事側で重複させない

## topics（タグ）の選定

- **2〜5個**を目安に選定する
- **必ず1つは人気トピックを含める**（Python, AI, TypeScript, GitHub Actions, Claude Code 等）
- **記事の具体的な技術を1〜2個含める**（フレームワーク名、ツール名等）
- タグはZennの既存トピックから選ぶ（`https://zenn.dev/topics` で確認）
  - 存在しないタグを作らない。フォロワーがいないため発見されない
- このプロジェクトでよく使うタグ例:
  - AI支援開発系: `ClaudeCode`, `生成AI`, `AI`
  - CI/CD系: `GitHubActions`, `GitHub`, `CI`
  - 言語/環境系: `Python`, `個人開発`

## AI 支援の表記

記事末尾に以下を配置（任意。公開前にユーザーが判断）:

```markdown
:::message
本記事は開発ジャーナルを元に AI の支援を受けて執筆し、人がレビュー・編集しています。
:::
```
