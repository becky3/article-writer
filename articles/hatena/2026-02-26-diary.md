## 指示文を書き直すうちに、仕組みに変えた

kuro-chan>>幸田姉さん、`ai-assistant` の `CLAUDE.md` に書いてある「最初に `README.md` を必ず読むこと」って一文、けっこう前から効いてないみたいで。

nee-san>>「効いてない」って、どういう意味で。

kuro-chan>>セッション開始直後に `README.md` を読まずに、いきなり質問へ答え始めることがあって。`python3` で動かないコマンドをそのまま叩いてエラーになるとか。

nee-san>>そりゃ事故ね。`README.md` に `uv run python` を使えって書いてあるんでしょ。

kuro-chan>>はい。なので「**必ず**」とか「他の作業より先に」とか強調表現を足そうかと思ったんですけど、それだとまた読まないときが出そうで。

nee-san>>太字にすれば読むって発想自体、人間相手でもだいぶ甘いわよ。マニュアルに赤字で書いても読まない人は読まない。

kuro-chan>>はい…で、よく考えたら `CLAUDE.md` は仕組みとして必ずセッション冒頭に流し込まれてるけど、その中に書かれた「`README.md` を読め」っていう一文は、読むかどうかが判断側に委ねられてる**指示テキスト**なんですよね。

nee-san>>仕組みで届くのと、指示で頼むの。階層がそもそも違う、と。

kuro-chan>>そうです。なので方針を変えて、SessionStart フックっていう仕組みのほうに `cat README.md` を仕込んで、セッション開始時に `README.md` の中身をまるごとシステムメッセージとして注入することにしました。

nee-san>>ああ、それなら判断は介在しないわね。届けたい情報は、届くルートに乗せる。

kuro-chan>>`CLAUDE.md` の記述も「読みなさい」じゃなくて「自動注入されます」に書き換えました。命令文じゃなくて、事実の説明に。

nee-san>>能動から受動。ちょっと哲学っぽいわね。

kuro-chan>>動作確認してみたら、注入されたあとは `uv run python` を自然に選んでくれるようになって。ランタイム情報が最初から目の前にあるんで、迷う余地がないというか。

nee-san>>「読め」と言うより「目の前に置く」のほうが速い。これ、社内マニュアルでも同じね。リンクで貼っても誰も飛ばないけど、本文に書いてあれば読むやつは読む。

kuro-chan>>ですね。重要な情報は、確実に届く仕組みに乗せる。これ、別のとこにも応用が効きそうです。

nee-san>>応用先を考え始めるの早すぎ。まずは今の対応で寝かせて、壊れないか見てから。

kuro-chan>>…はい。

## ドキュメント改修、丸３日かかったらしい

kuro-chan>>あと姉さん、`ai-assistant` の仕様書改革、今日で全部終わったみたいです。Phase 0 から Phase 6 まで全部。

nee-san>>あの 42 ファイル総点検のやつ？

kuro-chan>>はい。最終的に旧仕様書を新しい体系に移して、`specs-old/` と `specs-reform/` を丸ごと削除して。削除の差分が 9,720 行ありました。

nee-san>>1 万行近く消えた日って、なかなか見ないわね。

kuro-chan>>スタイルガイドを先に決めて、それに合わせてテンプレートを 11 種類用意して、個別の仕様書を 1 件ずつ移していく、っていう順番でした。テンプレートをいったん共通ファイルにまとめる方式にしたんですけど、途中で「いちいち見に行くのがだるい」っていう話になって、各テンプレート自己完結方式に戻したんですよね。

nee-san>>DRY を引っ込めるやつね。原則より使いやすさを取った、と。

kuro-chan>>はい。原則を守ったまま使いにくいテンプレートを残しても、結局誰も使わないので。

nee-san>>で、その締めの日に社長は何してたわけ？
