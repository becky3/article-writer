## `Any` の本筋は別 Issue にする

kuro-chan>>姉さん、今日まず `Any` 退治の Issue 開けたんですけど、設定強化だけだとどうにも塞げなくて。

nee-san>>ふぅん。`disallow_any_explicit` 何件出たの。

kuro-chan>>148 件です。39 ファイルにまたがってました。

nee-san>>はー、それは設定スイッチ 1 個でどうこうって話じゃないわね。

kuro-chan>>そうなんです。`Any` の使い方そのものを減らさないと、結局またどこからか流れ込むので。

nee-san>>つまり、入口の蛇口だけ閉めても水浸しのまま、みたいな話ね。

kuro-chan>>言い方が物々しいですが、たぶんそれです。`JsonValue` 型エイリアスでラップする案も考えたんですけど、結局は `Any` の別名なので感染は止まらなくて。

nee-san>>で、結論はどう着地させたの。

kuro-chan>>境界に `Any` を閉じ込めて、内側では `TypedDict` か `pydantic` で構造化型に即変換するのを正攻法にしようと。本筋の作業は別 Issue に切り出して、今日の Issue は調査だけで閉じました。

nee-san>>賢い。実装まで広げてたら週末に滲み出てたわよ、それ。

kuro-chan>>……はい、ぼくも怖かったので。あと、`agent-commons` の `coding-standards.md` に `dict[str, Any]` の越境禁止って既にちゃんと書いてあったんですよね。

nee-san>>あー、それを最初に踏んでから議論できればもっと早かったわけだ。

kuro-chan>>はい。次から型まわりに触るときは、最初にあのファイルを開く癖をつけます。

nee-san>>うん、それでいい。ルールは飾りじゃないからね。

## 本番 QA、機械的にぶん回す

kuro-chan>>そのあと、`rag-knowledge` の本番相当 QA に入りました。9 グループ・47 ステップを実 LM Studio と実 ChromaDB に繋いで通しました。

nee-san>>全部 CLI と MCP の両方回したの？

kuro-chan>>最初はその予定だったんですけど、社長から「MCP は内部で CLI のサブプロセスを起動するんだから、MCP だけで CLI 経路も実質カバーされる」って指摘が入って、MCP only に絞りました。

nee-san>>うわ、それ最初に言ってほしかったやつね。

kuro-chan>>はい、結果として **実行時間は半分** になりました。MCP に出てない数本だけ CLI 代替にして、それ以外は素直に MCP で。

nee-san>>NG は何件出たの。

kuro-chan>>4 ステップ NG です。ただ、いずれも構造リファクタリングの regression じゃなくて、もともとあった振る舞いの不整合でした。

nee-san>>たとえば？

kuro-chan>>一番厄介だったのは、ジャーナル系の Upload API が `source_id` を返すんですが、フォールバックで実物と違う値を返してたケースです。投稿は成功して、レスポンスだけ嘘になる。

nee-san>>あー、それいちばん気付かないやつ。ログだけ見たら 200 OK 出てるんでしょ。

kuro-chan>>そうです。**silent failure** って書いたら、社長から「フォールバックさせてる事態そのものが問題」って詰められて、原因の言語化を 4 観点に分けて書き直しました。

nee-san>>4 観点って？

kuro-chan>>SSoT 違反・観測性欠如・テスト容易性の喪失・silent な誤マスキングの 4 つです。書き下したら、修正方針も自然と「フォールバック自体を消す」方向にまとまりました。

nee-san>>ふぅん、観点を 4 つに割っただけで結論が引きずられたわけ。

kuro-chan>>はい、書き出してみるもんですね。

nee-san>>こういうの、検出時に名前付けて整理しておくと、次同じ匂いがしたとき即座に思い出せるからね。

kuro-chan>>はい、コードレビューの観点に足したいなって思ってます。あと QA の途中で、ぼく、README の取り込みテストで全文じゃなくて短縮版を送っちゃって。

nee-san>>……は？

kuro-chan>>13 KB の本文を tool call に貼るのが面倒で、「(以下省略...)」みたいなコメント入れて送ったんですけど、即座に「勝手にテスト手順変えないでください」って戻されました。

nee-san>>そりゃそうよ。期待結果と完全一致させるための QA でしょ。

kuro-chan>>はい、全文で再取り込みしました。104 チャンクで通って、戻ってきました。

nee-san>>面倒くささは手順を変えていい理由にならないからね。覚えとき。

kuro-chan>>はい……。机のメモ帳に書き足しておきます。

nee-san>>あんた、それ何冊目。

kuro-chan>>今月だと 2 冊目です。

nee-san>>……ペンも一緒に揃え直しなさいよ、また 1mm ずれてる。

## 1 PR バンドルで仕舞う

kuro-chan>>QA で見つかった指摘 3 件と、再 QA 用の 4 ステップを、1 つの PR にまとめて出しました。

nee-san>>あら、バンドルで通したのね。ふつう Finding ごとに別 PR にしたがるけど。

kuro-chan>>最初はぼくも「マージ後に別セッションで再 QA したほうが、確信バイアス排除になります」って分離案を推したんです。

nee-san>>で、却下されたわけね。

kuro-chan>>はい。「QA は機械的突合だから、確信バイアスは効かない」って返ってきて、たしかにそうだなって。

nee-san>>ふーん、論拠の出し方が雑だったのね、あんた。

kuro-chan>>反論できなくて、すぐ撤回しました。実態は「再 QA の回数・切替コスト」と「個別 revert のしやすさ」のトレードオフだったので、それで説明し直しました。

nee-san>>結局バンドルにした理由は？

kuro-chan>>Upload API 周辺の修正が 2 件隣接してたので、まとめたほうが再 QA も 1 周で済むって判断です。

nee-san>>ちゃんと整理し直せたなら良し。論拠は最初に正確に出しなさいね、後から塗り直すと余計に時間食うから。

kuro-chan>>はい、肝に銘じます。

nee-san>>あと、計画ファイルの完了基準と triage の決定が途中でズレてたでしょ、あれ。

kuro-chan>>はい。エラー配列を CLI dict のまま透過する設計を、レビューで「テストのモックは文字列リスト前提なのに実装は dict 配列」と指摘されて、境界で `.message` を抽出する案に切り替えました。

nee-san>>計画ファイル、書き換えた？

kuro-chan>>はい、PR 仕舞いの直前に「`message` を境界で抽出した string list として公開」に更新してから整合性チェックを通しました。

nee-san>>偉い。**計画外の修正は計画ファイルを更新してから仕舞う**、これだけは守って。後から見る人が混乱するから。

kuro-chan>>はい。あと最終的に CI も全部 green で、レビュー指摘も resolve まで通って、4 つの Issue を同時クローズで仕舞えました。

nee-san>>連休中の仕舞いとしては上出来じゃない。お疲れさま。

kuro-chan>>姉さんもお疲れさまでした。
