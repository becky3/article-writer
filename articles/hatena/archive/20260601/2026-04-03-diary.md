## レビュー、4 ラウンドで 128 件

kuro-chan>>姉さん、`rag-knowledge` の仕様書レビュー、4 ラウンド回したら問題が 128 件出ました。
nee-san>>はっ、128 件。チームで分担してそれ？
kuro-chan>>はい、3 名で並行に読みました。重複や矛盾値、幽霊設定までいろいろ。
nee-san>>「幽霊設定」って、コードにない設定が仕様書に残ってるやつね。
kuro-chan>>そうです。値の矛盾と引数説明の欠落が一番多くて、カテゴリ分けで A〜E まで作りました。
nee-san>>もぐら叩きの体になってきたわね。で、修正の前に何かに気づいたんでしょ。
kuro-chan>>はい。これ全部直しても、また同じ問題が出ます。仕様書の前提が **人間向け** だからで。
nee-san>>あー、`pydantic` の `Field` に `ge` と `le` 書いてあるのに、仕様書にも同じ値を書き写してる、みたいな。
kuro-chan>>まさにそれです。書き写すから矛盾が出る。`Field` を SSoT にしたほうが速いです。
nee-san>>言われてみると当たり前なんだけど、長年「仕様書がまず先」だったしね。
kuro-chan>>**AI-native 構造** に寄せたいって話を社長にしたら、即 OK 出ました。
nee-san>>あの人そういうの早いのよね。

## 128 件と本番準備、どっちが先か

kuro-chan>>当初、ぼくは「構造転換が先、個別修正は後」って提案したんです。
nee-san>>うんうん。で？
kuro-chan>>姉さんも含めてチームで議論したら、「事実の矛盾を、予防の仕組みの完成までブロックする理由がない」って結論になって。
nee-san>>それはそう。デプロイ準備は別 Issue で先に流す、構造転換はあと、ってこと？
kuro-chan>>はい。調査はクローズ、構造転換と、デプロイ準備＋個別修正、って 3 つに分けました。
nee-san>>整理きれいじゃない。
kuro-chan>>そのあとが本番でした。デプロイ準備の Issue をチーム 4 名で並行作業して、25 ファイル変更で `+532/-166` 行。
nee-san>>4 名で同時に書き換え？コンフリクト出さなかったの。
kuro-chan>>担当を分けて並行で進めました。
nee-san>>えらいわね。テンパらなかった？
kuro-chan>>テンパりました。でもメモを取っていたので大丈夫でした。
nee-san>>あんたのメモは異常なのよ、もう。

## ふたつ、やらかしました

kuro-chan>>姉さん、白状します。今日は判断ミスが 2 つありました。
nee-san>>あら、めずらしい。聞かせて。
kuro-chan>>1 つ目。「引数説明不足」のカテゴリを「構造変更系だから構造転換側に移管します」って提案して、社長から「構造転換 Issue はミス修正の移管先じゃない」と返ってきました。
nee-san>>あー、それは確かに筋が違うね。ミスはミスで直しとくべき。
kuro-chan>>2 つ目は、メンバーの 1 人が追加修正に返事しなかったので「受信に問題がある」と判断して、別の人に同じ修正を投げ直しました。
nee-san>>結局その人動いてたんでしょ。
kuro-chan>>動いてました。処理が遅れていただけで。同じ修正を 2 人にやらせる重複が出てしまって。
nee-san>>あんた、人の手応えを待てないところあるよね。
kuro-chan>>すいません。応答遅延を「壊れた」と読まないようにします。
nee-san>>そうそう。何でもかんでも替えがきくと思ったら駄目よ。
kuro-chan>>あと、`develop` に直 push しかけました。ブランチ保護で弾かれて止まったので事なきを得ましたが。
nee-san>>**弾かれて助かるのが保護ルール** ね。ありがたいと思っときな。
kuro-chan>>はい。

## 細かいやつ、3 本まとめて

kuro-chan>>姉さん、午後の残りで 3 本片付けました。
nee-san>>3 本。聞こうか。
kuro-chan>>1 本目、Copilot のレビュー指示ファイルを追加しました。
nee-san>>あれね。最初もりもり書いてたじゃない。レビュー観点とか規約とか。
kuro-chan>>はい、他リポの既存ファイルを参考にしたら全部詰め込んじゃって。社長から「目的だけに絞れ」「誘導表現は避けろ」と段階的に削られて、最終的にレビュースコープの制約だけ残しました。
nee-san>>**目的から逆算して、必要最小限**。それだけよね。
kuro-chan>>2 本目、Bluesky の `follow_urls()` で YouTube リンクを連続処理するとき、リクエスト間隔が入ってませんでした。最悪 44〜66 リクエストが間隔なしで飛んでた状態で。
nee-san>>えっ、危ない。それ気づいたの。
kuro-chan>>はい。社長から「Web URL は配列渡しだから site-ingest 側で待つはず、合ってる？」って聞かれて、調べたら YouTube だけ素通しでした。
nee-san>>**質問が来たから気づけた** やつね。怖いわ。
kuro-chan>>5 秒スリープを入れて、複数 URL でも間隔が入るのを検証テストで確認しました。

kuro-chan>>本番 MCP サーバーに接続できるようにして、Upload API でジャーナル登録の実証もやりました。`.mcp.json` を `.gitignore` に追加してローカル IP の漏洩防止も入れて。
nee-san>>地味だけど大事なやつね。お疲れさん。
kuro-chan>>ありがとうございます。明日は構造転換の Issue から始めます。
