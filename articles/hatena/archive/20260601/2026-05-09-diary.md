## `feed add` 1 分問題、ログから組み直す

kuro-chan>>姉さん、`ai-assistant` の `feed add` が本番で 1 分かかる件、ログを足して原因を追えるようにしました。
nee-san>>1 分。フリーズしてる扱いされるやつね。
kuro-chan>>はい。handler の完了ログがそもそも無くて、どこで詰まってるか分からなかったので。
nee-san>>あー、それで「お茶を濁す対応はやめろ」って社長から釘を刺されたんでしょ。
kuro-chan>>察しが良すぎます。タイムアウト + URL フォールバックでお茶を濁す案を一瞬考えたんですが、即却下されました。
nee-san>>そりゃそうよ。原因が見える状態を作るのが先。
kuro-chan>>なので handler / service / external の 3 階層に `start` / `complete` のログを入れる方針にしました。タイムスタンプ差分から時間が出るので、計測ロジックは入れずに済んでます。
nee-san>>シンプル方針、決めたとこ偉い。
kuro-chan>>CLI で叩くと 100ms、同じマシンの bot から叩くと 60 秒、っていう乖離も計測で見えました。
nee-san>>切り分けまで持っていけたなら大成功よ。
kuro-chan>>その代わりレビュー指摘が 11 件来て、全部 resolve するのに半日かかりました…。
nee-san>>観測性入れた日にレビュー観測性も浴びる、って構図ね。

## `.env.example` のプレースホルダ、CI で OS の壁にぶつかる

kuro-chan>>姉さん、もう 1 件あって。`.env.example` の値が 6 個空欄だったので、汎用ダミーを入れたんですが。
nee-san>>パス系は Win 形式にしたんでしょ。
kuro-chan>>はい、`C:/path/to/...` で。`Path("C:/...").is_absolute()` が Win では True なので。
nee-san>>……それ Linux で False になるやつ。
kuro-chan>>CI で見事に落ちました。
nee-san>>うちの CI、Linux で動いてるんだから先に気付けなさいよ。
kuro-chan>>ローカル Win では全部 pass してて、push したら CI が真っ赤になって。
nee-san>>「全部 pass」って一番油断するパターン。
kuro-chan>>テスト側で OS 分岐の monkeypatch を足して、ついでに「`.env.example` に空欄が紛れ込んだら落ちる」ガードレールテストも入れました。
nee-san>>後で同じことを繰り返さない仕組みまで入れたなら、それは偉い。
kuro-chan>>Copilot がクロスプラットフォーム起動シナリオを心配してくれたんですが、`triage` で「Win 主環境前提・テンプレートは編集前提」と確定済みだったので、全部 ❌ で返しました。
nee-san>>前提が事前に揃ってると、Copilot 対応も即決できて気持ちいいね。
