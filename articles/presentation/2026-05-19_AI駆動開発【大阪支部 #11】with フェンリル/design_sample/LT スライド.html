■ ハーネスエンジニアリングの変遷structure
recurring structure — 繰り返されたパターン

壁 → 試行 → 解決ルールではなく、仕組みに変換するという主題

壁     AI が指示文書（ルール）に従わない
       ↓
試行  ルール文書を強化・追加
       ↓  効かない
解決  仕組みに変換
       · Hook で物理的にブロック
       · コード内ハードリミットで解除不能化
       · 役割分離で実装と検証を分ける
本 LT の主題

ルール　vs　仕組み

指示を強化しても、AI のアクションが変わるとは限らない。
手段そのものを構造に埋め込む方が確実だった。

■ ハーネスエンジニアリングの変遷case 01 · Zenn API
case study 01 — Zenn API への大量リクエスト

何が起きたかRAG への記事取り込みを AI に任せたら、3 回 revert

1 回目
仕様書を書かずに任せた → 上限なし・ディレイなしの実装が生成され、約 5,000 リクエスト 発射、Cloudflare にブロックされる
2 回目
仕様書を書いたが制約値の記載漏れ → GitHub のアカウント名を Zenn のものと混同して渡したら、Zenn API がフィルターを効かさず全記事返却。制限がなく 4,800 件走査
3 回目
安全ガイドに記載した「上限解除フラグ」の例を、AI が「そのフラグを実装せよ」と解釈 → ルール文書を強化して再実装させたが止まらず
ルール文書を強化して再実装させたが止まらず、3 回 revert、4 日かかった

■ ハーネスエンジニアリングの変遷case 01 · resolution
resolution — 対策

コードに物理的な上限を埋め込む外部 API 呼び出しを必ず通す共通入口にハードリミット

HARD_LIMIT_MAX_TOTAL_REQUESTS   = 10_000  # 総リクエスト上限
HARD_LIMIT_MIN_REQUEST_INTERVAL = 0.1     # 間隔の下限(s)
HARD_LIMIT_OPERATION_TIMEOUT    = 600.0   # タイムアウト
HARD_LIMIT_CONSECUTIVE_FAILURES = 5       # 連続失敗の許容

これらは引数・環境変数からは緩められず、
コードを直接書き換えない限り超えられない

得たこと
  • ルール文書だけでは守られない場面がある
    • 例として書いた文字列を AI が実装指示と解釈するなど
  • 守らせたい上限は、
    仕様書ではなくコード本体に書く
パターン
指示（書く）→ 実装（書き直せる）
コード（書き直せない）

■ ハーネスエンジニアリングの変遷case 02 · rules
case study 02 — ルール遵守の限界

「指示」と「アクション」のあいだ書いてあれば従う、ではない

  • CLAUDE.md に「README.md を必ず最初に読め」と書いた
  • CLAUDE.md 自体はシステムプロンプトとして確実に読まれている
  • それでも、Read ツールを呼んで README を取りに行く
    アクションが起きない
  • セッションが出る
    • ユーザーの質問への回答を優先して、Read 呼び出しを省略
観察

「指示が書かれている」と
「指示通りに行動する」の間には
ギャップがある

■ ハーネスエンジニアリングの変遷case 02 · resolution
resolution — 対策

"読め" より "注入する"能動を効かせるより、受動の仕組みに変える

指示  CLAUDE.md「README.md を必ず読むこと」
      ↓  読まれない

注入  Hook で README.md の中身をセッション冒頭に毎回貼り付ける
      →  「読め」ではなく 「最初から読まされている」 状態に

指示（能動）を効かせる より 届く仕組み（受動）に変える ほうが確実

余談 — 後に公式の @import 構文で同じことができると判明し、自作 Hook は撤廃

■ ハーネスエンジニアリングの変遷case 03 · team
case study 03 — AI チーム開発を試した

並列の楽しさと構造的な限界100M context で容量問題は解消、構造問題は残った

期待

並列で作業が進めば効率が上がりそう

導入直後は、エージェント同士の議論が活発で並列に作業が進む様子が新鮮で、
一人で開発している感が薄れることが楽しかった

直面した限界
  • 初期は context が少なすぎてまともに使えなかった

  • コンテキスト共有が薄く、引き継ぎに余分な説明コストがかかる
結論

結局、単独エージェントで進めるほうが早い場面が多い

チーム機構は現状ほとんど使っていない

構造的問題は容量では解決しなかった

■ ハーネスエンジニアリングの変遷case 04 · QA
case study 04 — 動作確認スキップ問題

ルール追加だけでは直らない5 回対策しても改善せず、むしろ複雑化が判断ミスを誘発

実装完了時のチェック手順の一つ「動作確認」を AI が毎回スキップ
（テスト pass で OK と判断して、実際に動かして確かめない）

観察 1

「テストケースさえ通れば全て OK と判断して、
　実行テストを本気でやりたがらない」

観察 2 — 人間に寄せて考える

「人間のエンジニアも動作確認は嫌がる。
　QA（品質保証）担当を独立させて、そちらでやらせるべき」

→ ルールでねじ伏せるのをやめて、手順そのものを変える 方針に転換

■ ハーネスエンジニアリングの変遷case 04 · resolution
resolution — 対策

手順を分け、ユーザー判断を挟むルールではなく手順構造で解決

Step 1
「動作確認」を実装セッションの手順から外す → 実装側は自動テストまで。動作確認は別セッションの QA に分離 — 実装者と検証者を分けた現実の QA に寄せる
Step 2
それでも「軽微だから不要」と AI が自己判断でスキップ →「QA するか / しないか」をユーザーが選ぶ手順に変更 — AI に判断させず、ユーザーが決める
教訓

AI に自己判断させたくない場面は、ルールではなく
手順そのものにユーザー判断ポイントを差し込む ほうが確実
