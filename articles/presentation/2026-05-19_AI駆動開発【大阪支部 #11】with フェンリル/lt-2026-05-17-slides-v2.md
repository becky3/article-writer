### [4-1] ★ Zenn API 5,000 リクエスト事件 → ConstrainedClient（45 秒）

#### 本文

```
事件: 仕様書なしで auto-implement に踏み切った

  自動実装ラベル → AI が discover() を生成
  → ページネーション上限なし・ディレイなし
  → Zenn API に 5,000 リクエスト
  → Cloudflare に IPv6 だけブロック

  3 波 revert、3〜4 日溶かす

打ち手: コードレベルで解除不能なハードリミット

  HARD_LIMIT_MAX_TOTAL_REQUESTS   = 10_000
  HARD_LIMIT_MIN_REQUEST_INTERVAL = 0.1
  HARD_LIMIT_OPERATION_TIMEOUT    = 600.0
  HARD_LIMIT_CONSECUTIVE_FAILURES = 5

  → 設定・引数・環境変数のいかなる手段でも引き上げ不可
```

#### ねらい

- 映え最強のエピソード
- 「自動化が回り始めた瞬間が最も危ない」という構図
- 「**仕様駆動開発は、仕様書の作成部分を放棄してはいけない**」（user の反省）
- ConstrainedClient の設計思想: コード内定数で解除不能なハードリミット
- 教訓キーフレーズ: 「**ルールの模範例は実装指示として読まれる**」「**上限値ではなく許容範囲**」「**デフォルト値 ≠ 上限値**」

#### 画像案

- IPv6 だけブロックされた図（curl -4 OK / curl -6 NG）
- ConstrainedClient のコード抜粋（4 つの HARD_LIMIT 定数）
- 「3 波 revert / 3〜4 日溶かす」を数字でハイライト

## [5] セッション中の workflow + スキル紹介（1 分）

### 本文

```
Issue を 1 件片付ける 9 ステップ

 1. 復帰        /restore        前回ジャーナルから文脈復元
 2. 着手        /start-work     Issue 取得・worktree 作成
 3. 計画        /plan-work      影響範囲調査・完了基準合意
                                ※ Hook が計画なし編集を block
 4. 確認        /triage         複数項目は確認表ファイル方式
 5. レビュー    code-reviewer + doc-reviewer 並行実行
 6. 指摘対応    /triage         要修正は自動 / 要相談は確認表
 7. 締め        /auto-finalize  test → push → PR → 監視ループ
                                ※ gh pr create は Hook で経路ガード
 8. マージ後    /merged         チーム解散・ブランチ削除
 9. 引継ぎ      /handoff        ジャーナル化・課題 Issue 化
                                └→ 次サイクルの /restore に繋がる

デフォルトスキルで定着したのは /loop だけ
効いたのは Skill 機構そのもの
→ SKILL.md を書くだけで自分の運用が「型」として残る
```

### ねらい

- 9 ステップを 1 サイクルとして俯瞰
- スキルが「単発便利ツール」ではなく「1 つの運用システム」として結合
- `/handoff` で書いたジャーナルが次サイクルの `/restore` の入力になる**ループ構造**
- Hook 介入箇所（ステップ 3 後 / ステップ 7 内）を強調
- デフォルトスキルは `/loop` だけ。本命は Skill 機構そのもの
- 聴衆が「自分でもこの workflow を組みたい」と思える具体度

### 画像案

- 9 ステップを円環状に配置（1 → 2 → … → 9 → 1 のループ）
- 9（handoff）→ 1（restore）を太い矢印でループ強調
- Hook 介入箇所を赤い縦線でハイライト
