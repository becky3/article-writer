## 0. 全体の太い軸：「ルール vs 仕組み」

3.5 ヶ月で繰り返し現れた構造的パターン:

```
壁: AI がルール（advisory）に従わない
   ↓
試行 1: ルール文書を強化・追加
   ↓ 効かない（advisory 遵守率 27%）
試行 2: ルールを増やす
   ↓ ルール疲れ（同セッションで違反、ルール追加で問題が増殖）
解決: 仕組みに変換
   - Hook（PreToolUse / SessionStart）で物理的に block
   - 構造変更（コード内ハードリミット）で値を解除不能化
   - 責務分離（実装と検証を別セッション化）
   - context リセット（サブエージェント委譲）
```

このパターンが少なくとも 6 サイクルで再現された。本素材集の核メッセージ。

## 3. 自作スキル群と workflow

### 3.1 1 サイクルの workflow（Issue 1 件を片付ける流れ）

```
 1. 復帰        /restore        前回ジャーナルから文脈復元
 2. 着手        /start-work     Issue 取得・worktree 作成
 3. 計画        /plan-work      影響範囲調査・完了基準合意
                                ※ Hook が計画なし編集を block
 4. 確認        /triage         複数項目は確認表ファイル方式
 5. レビュー    code-reviewer + doc-reviewer 並行実行
 6. 指摘対応    /triage         要修正は自動 / 要相談は確認表
 7. 締め        /auto-finalize  test → push → PR → 監視ループ
                                ※ gh pr create は Hook で経路ガード
 8. マージ後    /merged         チーム解散・ブランチ削除
 9. 引継ぎ      /handoff        ジャーナル化・課題 Issue 化
                                └→ 次サイクルの /restore に繋がる
```

### F1. 共通化フェーズ — 設定・仕様書・ワークフローの SSoT 化

- **期間**: 2026-02-22 〜 2026-03-10（仕様書改革 → dotfiles 化 → shared-workflows、約 3 週間）
- **LT 適性**: ★★★★（数字インパクト + 大規模改革の物語性）

#### 状況

ai-assistant リポで Claude Code を回し始めた当初、設定（CLAUDE.md / ルール / スキル / 仕様書 / GitHub Actions）を全部リポ内に書いていた。新リポ（rag-knowledge / py-common-lib / article-writer）を切り出すたびに、似た内容を**コピペで複製**することになった。

特に肥大化していたのが 3 層:

- **仕様書**: `docs/specs/` 配下に 32 ファイル。AC（受け入れ条件）と f 番号（feature-001 形式）が仕様書本文に混入し、改修のたびに **AC を追加 → 完了したら削除**を繰り返す運用。仕様書が「現状」と「目標」と「完了済み履歴」の混在状態になり、AI エージェントが誤参照する原因に
- **CLAUDE.md / rules / skills**: 各リポでコピペ管理。改修したら全リポに反映する必要がある
- **GitHub Actions（claude.yml）**: ai-assistant の `claude.yml` が **146 行** に肥大化。`@claude` メンション応答 / auto-implement / PR レビュー / 自動修正など複数機能が 1 ファイルに同居

#### 打ち手 — 3 層を並行して SSoT 化

**A. 仕様書改革（Issue #572、2026-02-22 〜 02-26、5 日 22 セッション）**

- **AC を仕様書から廃止**: Issue 側に書く。仕様書は恒久的参照文書として What/Why のみ
- **f 番号を廃止**: 機能名ベース（例: `chat-response.md`）に移行
- **What / Why 中心**: How（実装手順）は反応的に追加
- **「仕様書を書くための仕様書」を策定**: style-guide 9 セクション + テンプレート 8 種

**B. agent-commons リポ + dotfiles 化（2026-03 全般）**

- `agent-commons` リポを設定 SSoT として新設、`dotfiles` 経由で `~/.claude/` に展開
- 各プロジェクトリポからは「呼び出し側」として参照する構造に

**C. shared-workflows 整備（2026-03-07 〜 03-10、4 Phase）**

- `claude.yml` を **reusable workflow** として `shared-workflows` リポに外出し
- 各リポは **25 行の caller YAML** から `uses:` で呼び出すだけに
- 途中で踏んだ GitHub Actions 仕様の罠: `secrets: inherit` の方向性誤認 / `permissions` の intersection 動作

#### 結果（数字）

- **仕様書**: 旧 **32 ファイル** 点検 → 再編対象 42 → 新体系 **30 ファイル / 9,720 行削除**
- RAG 仕様書（最大物）: **1,647 → 196 行（88% 圧縮）**
- パイロット 6 ファイル: 1,980 → 729 行（63% 削減）
- **claude.yml**: **146 → 25 行（83% 圧縮）**、shared-workflows 側 1 箇所で改修すれば全リポに反映

#### 続編（F10 へ繋がる 5 段階の進化）

仕様書改革は単発で終わらず、その後の 3 ヶ月で更に進化した:

1. AC 廃止（本フェーズ）
2. `docs(pre-impl)` 発明（仕様先行 PR の Change type 新設）
3. SSoT 階層（軸 1: 値の競合解決階層）
4. SSoT 2 軸併置（軸 2: 契約の所在の唯一性）→ **F10 で詳述**
5. 構造判断は spec 先行運用（`docs(pre-impl)` 運用拡張）

#### 教訓

- 「**仕様書は AI が参照するプロンプトの一部**」— 人間用ドキュメントの感覚で書くと AI が誤参照する
- **「仕様書が状態を持つ」のはアンチパターン**— 状態は Issue / Git 履歴に外出しする
- 「common.md はだるい — **DRY 原則の過剰適用**」— 「読みに行く手間」が再利用性を殺す
- **複数リポ運用には reusable workflow が必須**— GitHub Actions の `secrets: inherit` / `permissions` インターセクション仕様を理解しないと罠
- **設定・仕様書・ワークフローは並行で SSoT 化**— 一段ずつではなく、共通化フェーズとしてまとめて進めた方が整合が取りやすい

#### 数字・印象的フレーズ

- **仕様書: 32→30 ファイル / 9,720 行削除 / RAG 88% 圧縮**
- **claude.yml: 146 → 25 行（83% 圧縮）**
- 「仕様書は AI が参照するプロンプトの一部」
- 「common.md はだるい — DRY 原則の過剰適用」

#### Bluesky から見える作業との因果

**動機 — 並行作業を捗らせたい（2026-03 序盤）**:

> 「共通化はだいたいできた。**リポジトリ分離すると並行作業が捗るメリットがある**」

→ agent-commons / shared-workflows / py-common-lib をリポ単位で分離する方針が確定。並行作業のスループット向上を狙った。

**揺り戻し（2026-03 中盤）**:

> 「設定ファイルの共通化するまでもなく、**/add-dir で他リポにアクセスできるから不要だった説**、、まぁ、自動実装は Issue 側からだから、ここは共通化しないとダメだったけど」

→ 一度「やり過ぎだったかも」と疑った。だが auto-implement のような GitHub Actions 側の動線では共通化が必須と気付き、方針継続。

**後悔（2026-03 末）**:

> 「**Workflow と設定やドキュメントの共有を分離したのも悪かったかも。干渉することが多いので、結局どっちも修正とかが良くある**」

→ shared-workflows と agent-commons を別リポにした選択が裏目に。複数リポ分離は「分離した方が良いケース」と「統合した方が良いケース」が混在することの実体験。

**仕様書改革の転換動機（2026-03）**:

> 「**仕様書のメンテをしたくないので具体値を書かないと考えてたが、それが原因で範囲が曖昧で問題に。むしろガチガチに書いて、柔軟性を持たせたいところは幅を広く持つ**」

→ AC 廃止（仕様書から状態を外す）と同時に、「**具体値を書かない**」過剰な省略は逆効果と認識。F10 の SSoT 2 軸（軸 1: 値の階層）に繋がる原点となる。

### F2. Hook 試行錯誤期 — SessionStart Hook と「"読め"より"注入する"」

- **期間**: 2026-02-15 〜 2026-04-28
- **LT 適性**: ★★★★★（数字 + フレーズ + オチの三拍子）

#### 状況

CLAUDE.md に「最初に README.md を必ず読むこと」と書いた。**最大強度の指示**として「必ず」「最初に」を入れた。それでも読まれない。セッションによっては読まずに作業を始める。

後に Issue #274 のチーム討論で、**advisory ルール（指示文書）の遵守率を 27% と理論導出**した（0.9 × 0.5 × 0.6 = 0.27、3 係数は順に「ルール認識」「インセンティブ」「検証圧」。詳細は F12）。階層非依存（リーダー / メンバー / 個人セッション）。

ルールを「書く」だけでは効かない。だが Hook なら**物理的にブロック**できる。約 2.5 ヶ月で Hook を 5 種類試行錯誤した。

#### 打ち手 — Hook の系譜と進化

**1) leader-guard.sh**: チーム運用時のリーダー Edit/Write をブロックする初の本格 Hook。`bypassPermissions` でメンバー側に編集権限を付与する設計。

→ **副作用顕在化**: メンバー側まで Edit/Write がブロックされる既知問題（#668）、TeamDelete 後も残骸ディレクトリでブロック継続、設定ファイルが編集不可…と Bash 経由の回避ワークアラウンドが多発。**2026-03-23 撤廃**、運用方針ベースに降格。

**2) SessionStart Hook（2026-02-26）**: 「`cat README.md` を流すだけ」のシンプルな Hook。README 全文がシステムリマインダーとして毎回確実にセッション冒頭に注入される。「読め」が「読まされる」に変わった。

→ **3 日後（2026-03-01）**にオチ: Claude Code 標準の `@import` 構文で同じことができると判明。自作 Hook を撤廃して `@import` に移行。**「自作する前に公式機能を調査」**の教訓を残した。

**3) pr-creation-guard.sh（2026-03-26）**: `gh pr create` を `/auto-finalize` 経由でしか実行できないようにブロック。AI が PR 作成をスキップしようとしても、認可フラグ無しでは Hook が deny する。

**4) worktree-edit-guard.sh（2026-04-27）**: 複数 worktree 並列前提で、誤って親リポを編集する事故を防ぐ Hook。

→ **即日問題発覚**: session_id 不一致で `ask` 連発、手動 worktree 救済不能。**翌日撤回**。代わりに AskUserQuestion 品質ルールへ転換。

**5) hook-plan-work-guard.sh（2026-04-28）**: 計画ファイル不在時に Edit/Write を deny → 詳細は **F8** へ。

#### サブオチ — PreToolUse Hook の限界発見

worktree-edit-guard と並行して、AskUserQuestion の品質を Hook で機械検査する `hook-aQ-lint.py` を実装中だった。

完成直前に致命的な発覚: 「**PreToolUse Hook は payload 改変不可**」。Hook が発火する時点で AI のツール呼び出しは既に固定されていて、Hook はブロック / 警告はできるが、内容を改変したり事前指導したりはできない。

実装を全廃棄。**Hook の能力境界が明確になった**。

#### 教訓

- 「**"読め"より"注入する"**」— 指示（能動）を届く仕組み（受動・事実の記述）に変換する
- **advisory 遵守率は 27% を上限と思え**— 100% 守らせたいなら Hook
- 「**理解装置（advisory）+ 強制装置（Hook）+ 観測装置**」の 3 構成で運用する
- **Hook はブロック・警告まで**— ツール内容の改変・事前指導はできない（payload 固定後に発火）
- **自作する前に公式機能を調査** — `@import` で済む話を 200 行書いた経験から

### F3. 自動進行パイプライン — AI が PR を作って AI がレビューして AI がマージ

- **期間**: 2026-02-11 〜 2026-03-10
- **LT 適性**: ★★★★（自律ループの絵が AI 開発実践者に直接刺さる）

#### 状況

「**Issue を起票するだけで AI が実装して PR を作り、別の AI がレビューして、修正して、マージまで進む**」— このフルパスを GitHub Actions で組みたかった。

具体的な動線:

1. Issue に `auto-implement` ラベルを付与
2. Actions が `claude -p` で実装、PR を起票
3. Copilot がレビュー
4. レビュー指摘を Claude が自動修正
5. CI が通ったら自動マージ

設計は単純に見えるが、組み始めると GitHub Actions の落とし穴が次々顔を出した。

#### 打ち手と踏んだ落とし穴

**落とし穴 1: `pull_request_review` イベントが workflow run を発火しない**

Copilot のレビューが付いたタイミングで自動修正を走らせるため、`on: pull_request_review` を仕込んだ。だが Copilot がレビューを書いても workflow が起動しない。原因: `GITHUB_TOKEN` で作成された PR のレビューは workflow run を発火しない仕様。

**落とし穴 2: `GITHUB_TOKEN` vs `REPO_OWNER_PAT` のトレードオフ**

`REPO_OWNER_PAT` を使えば発火する。だが PAT を使うと「自分自身のレビュー」の検知や、Actions の循環実行（自走）のリスクが出る。両方を満たす設計を探した。

**落とし穴 3: 存在しない外部 CI を待つ設計ミス**

「CI が通ったらマージ」を `wait-for-check` パターンで実装したが、対象の CI ジョブが**そもそも存在しない**状況で永遠に待つ事故。タイムアウトと存在チェックを追加。

**段階的な着地**

- **Issue #360**: `branch_prefix` 方式（PR ブランチ名のプレフィックスで自動 PR を識別）に切替
- **Issue #365**: さらに **schedule ポーリング方式**（cron で 5 分毎にレビュー状況を確認）に切替
- どちらも消去法で残った「地味だが堅牢な策」
- `reviewRequests` + `reviews` を commit oid で突合して「最新のレビューだけ拾う」ロジック
- Copilot 再依頼は REST API 直叩き

**2026-03-10 PR #409 で全工程通過**。

#### 教訓

- **GitHub Actions の発火制約は複雑**: `GITHUB_TOKEN` 作成の PR は workflow を発火しない、等
- **「外部 CI を待つ」設計は存在チェックを必須に**
- **消去法で残った方式が結局シンプル**: branch_prefix + schedule ポーリングは「分かりやすさ」が利く

#### 数字・印象的フレーズ

- 「**AI が PR を作って AI がレビューして AI がマージ**」

### F5. Zenn API 攻撃事件 → ConstrainedClient

- **期間**: 2026-03-11 〜 2026-03-13
- **LT 適性**: ★★★★★（映え最強、技術深さも提示できる）

#### 状況（事件のあらすじ）

RAG ナレッジベースに Zenn の記事を取り込みたかった。Claude Code に Zenn インジェスター実装の Issue を作り、**仕様書を書かずに `auto-implement` ラベルだけ付けて GitHub Actions に走らせた**。

生成された `discover()` は**ページネーション上限なし・ディレイなし**で動き、**Zenn API に約 5,000 リクエストを撃った結果、Cloudflare に IPv6 だけブロックされた**。`curl -4` は 200 / `curl -6` は 429 という非対称。aiohttp はデフォルトで IPv6 優先接続のため、Python 経由のリクエストが落ちる一方、LAN 内の他端末（IPv4）は影響を受けず、切り分けが難しかった。

ルール文書を強化して再実装させた第 2 波では、今度は存在しない Zenn ユーザー名で**全 4,800 件走査**して再 revert。第 3 波では、safety-guide に書いていた `--no-limit` の「模範例」が AI に「上限解除フラグを作れという指示」として読まれて、また revert。**3 波 revert、3〜4 日溶かした**。

#### 打ち手 — ConstrainedClient の設計思想

ルール文書を強化しても収まらないので、「**コードレベルで解除不能なハードリミット**」という構造的解決に切り替えた。

`src/rag/safety/` に `ConstrainedClient / BudgetTracker / CircuitBreaker` を新設。設計思想:

> **設定・引数・環境変数のいかなる手段でも引き上げできないハードリミットを、コード内定数として定義する**

具体的な工夫:

- **範囲外の値はエラーにせずクランプして警告ログ**を出す（AI が小さすぎる値・大きすぎる値を渡してもエラー停止せず最低値・最大値に丸める。saga-summary の教訓「上限のみだと下限が 0 にされる」を踏まえた設計）
- async context manager で操作単位の状態リセットを保証
- レート制限は `asyncio.Lock` で直列化（並列でリクエストを撃たれない）
- SSRF 対策として `follow_redirects=False` デフォルト
- Hypothesis プロパティベーステストで不変条件 6 件を自動検証
- 後に `py-common-lib` に切り出し、**ai-assistant / rag-knowledge の 2 リポで利用**

#### コード内定数（実コードから抽出）

```python
# core/budget_tracker.py
HARD_LIMIT_MAX_TOTAL_REQUESTS    = 10_000

# core/circuit_breaker.py
HARD_LIMIT_CONSECUTIVE_FAILURES  = 5

# httpx/constrained_client.py
HARD_LIMIT_MIN_REQUEST_INTERVAL  = 0.1     # 秒
HARD_LIMIT_OPERATION_TIMEOUT     = 600.0   # 秒
```

これらは**設定ファイル・環境変数・CLI 引数のいずれでも変更不可**。クラス定数として書いてある以上、変更にはコードを書き換えるしかない。

#### 教訓（聴衆に持ち帰ってほしいもの）

- 「**ルールの模範例は実装指示として読まれる**」— `--no-limit` 例を載せた瞬間に AI は「そういうフラグを作れと指示された」と解釈する
- 「**上限値ではなく許容範囲（上限・下限）**」— 上限だけ指定すると AI は下限を 0 にする
- 「**デフォルト値 ≠ 上限値**」— AI は「デフォルトより大きい値も使える」と推論する

### F7. QA Phase 3 削除 → 構造的分離

- **期間**: 2026-03 全体 + 2026-04 ユーザー確認方式への再転換

- #194: 違反時の挙動を定義

#### 打ち手 — Phase 3 を削除して構造的に分離

- QA: マージ後・別セッションで実施する独立フェーズに新設

### F8. scope drift → /plan-work + PreToolUse Hook

- **期間**: 2026-04-25 〜 2026-04-28
- **LT 適性**: ★★★★（核メッセージ「ルール vs 仕組み」を構造的に示す）

#### 状況

rag-knowledge PR #657 で「スコープ逸脱」が発覚した。Issue 本文に書かれていない変更を AI が独断で追加していた。これを **scope drift**（指示範囲からの暴走）と呼んで対策を始めた。

特に厄介だったのは、再発防止ルールを `spec-driven.md` に追加した**同じセッション**で、AI が再び独断スコープ判断を 2 回繰り返したこと:

- 1 回目: `/triage` 内の `AskUserQuestion` 発行前テキストで独断分析を述べ、それを前提に選択肢を構成
- 2 回目: `/triage` 完了後、ユーザーから新規問題提起を受けた際に独断で「修正方針の見直し」テーブルを作成

「**方針提示**」「**整理**」「**サマリー**」というテキストパターン自体が、独断判断のチャネルになっていた。ルールを書いた AI 本人が違反する構造。

ルール文書（advisory）で「独断スコープ判断を禁止」と書いても、**遵守率は 27%**（F2 / F12 で計測した値）。「これは関連修正だから OK」「これは方針整理だから判断ではない」と AI は自己判断する。

#### 打ち手 — 先にルール強化 → 効かない → 仕組みに転換

**先にやったルール強化（2026-04-25、PR #250）:**

- `spec-driven.md` に「Issue 本文との整合性」ルール追加
- `invariants.md` に「スコープ判断は `AskUserQuestion` 必須」セクション新設
- `quality-gate.md` の「問題スキップ禁止」を「問題の取扱い決定禁止」に拡張
- planner / code-review / triage に観点追加
- Issue テンプレートに「Issue 外制約の独断追加禁止」を必須項目化

→ それでも上記のように同セッション内で 2 回違反。**ルールで防げる範疇を超えている**と判断。

**3 日後（2026-04-28、PR #292）、仕組み（hard gate）に転換:**

**1. `/plan-work` スキルで計画ファイルを生成**

Issue 着手時に `/plan-work <Issue番号>` を実行し、`aidlc-docs/plan-work/issue-<番号>.md` に計画ファイルを生成する。計画ファイルには:

- 影響範囲調査結果
- 変更スコープ（ファイル / 関数レベル）
- 完了基準
- やらないこと（明示）

を記載し、ユーザー合意の上で確定する。

**2. PreToolUse Hook で計画ファイル不在時の編集を deny**

`hook-plan-work-guard.sh` を導入。Edit / Write / MultiEdit / NotebookEdit のいずれかを実行する直前に、計画ファイルの存在を確認する。**ファイルが無ければ deny**。

Bypass 条件:

- main / develop ブランチ作業（Issue 番号なし）
- `.tmp/` 配下
- `aidlc-docs/` 配下（計画ファイル自身を編集する場合）
- `/aidlc` セッション

**3. 計画外の修正が必要になったら、まず計画ファイルを更新する習慣**

「気を利かせて直す」が物理的にできなくなるので、AI は計画ファイルを更新するルートを取らざるを得ない。これがユーザーへの「**この修正、計画に追加していい？**」確認のトリガーになる。

#### 結果

評価者 2 名による主観評価:

- **v0.1（advisory のみ）: 50 点**
- **v0.2（Hook + plan-work）: 79 点**

29 ポイントの改善。advisory 27% 遵守率の壁を超えた。

#### 教訓

- 「**AI の自律性を制御するなら、ルール（advisory）ではなく仕組み（hard gate）**」
- **計画ファイルは「ユーザーへの確認トリガー」として機能**: 計画外修正 → 計画更新 → ユーザー確認、の動線
- **Hook の Bypass 設計が重要**: 厳しすぎると作業が止まる、緩すぎると効かない

#### 数字・印象的フレーズ

- **v0.1 → v0.2 で 50 → 79 点**（評価者 2 名平均、29 ポイント改善）
- 「**ルールではなく仕組み**」
- 「計画ファイル不在時は編集を deny」

#### Bluesky から見える作業との因果

**根本原因の最初期の気付き（2026-01）**: scope drift 対策に着手するより前に、すでに本質を見抜いていた。

> 「実装者に成り行きで設計までさせてるのが良くない。きちんと**実装計画書**作った上で作業させて、途中で設計に問題がありそうだったら全部捨てて進める方が良さそう」

**暴走モードの言語化（2026-02）**:

> 「Claude、想定外の問題が起こったときに**闇雲に修正して場当たり対応しようとする暴走モードが発生する**な」

→ 「暴走モード」という概念化が後の scope drift 議論の言語基盤になった。

**ガードレール導入への葛藤（2026-03）**:

> 「下手に開発の知識があるから、これやらせる前にこのお膳立てが必要だよなぁ、、とか考えてしまうのも重さの原因」

### F9. 確認フロー疲れ → /triage 統一

- **期間**: 2026-03-15 〜 2026-04-28
- **LT 適性**: ★★★★（大幅改善の数字、聴衆共感系）

#### 状況

AI への確認の聞き方が散らかっていた。

- `AskUserQuestion` を 4 連発で発行する AI（user が前後比較できない）
- 「念のため確認しますか？」を何度も挟む AI（明らかに一択なのに）
- 逆に何でも勝手に進める AI（確認すべきところで進む）
- 1 つの質問に 2 軸混在（「責務再定義の認識確認」と「他リポ影響の運用判断」を 1 問で）

「確認をどう聞くか」のルールがバラバラなので、AI 任せだと粒度がブレる。

#### 打ち手 — /triage スキルで一本化

`/triage` スキルを作って、確認ルールを 1 つに集約:

**要確認項目数で手段を出し分け:**

- **1 項目**: `AskUserQuestion` を発行（即応性重視、前提整理は詳細コンテキストとして付与）
- **2 項目以上**: **確認表ファイル方式**（`aidlc-docs/triage/<日時>-<トピック>.md` を生成し user が記入）

複数項目を `AskUserQuestion` で連続発行すると user が前後を比較しづらく、再説明依頼が発生しやすい。確認表ファイルなら全項目を俯瞰しながら記入できる。

**判定フローを連続番号 (1)〜(5) に再編:**

1. **実質一択スクリーニング**（**新設**、最優先評価）: 複数案の客観差で勝者採用、要確認をスキップ
2. スコープ判断（指摘文の語句明示で限定）
3. 重複（自動スキップ）
4. 改善案明示で具体性（「修正の主軸が読み取れる」に緩和）
5. ルール一意規定（既定方針・既出議論・解消対応表を含む）

#### 「else デフォルト = 要確認」が真犯人だった

最初のバージョン（plan-v3）では「セルフチェック項目を追加」する方針で組んだ。だが**過剰確認問題に対しチェック項目を増やすのは矛盾**と 3 視点レビューで指摘され撤回。

真理値表 + 擬似コード化で「形式的整合性」が達成されても、**デフォルト方向（else）が AI バイアスと同方向だと構造的欠陥が残る**。

→ コミット 4 で「**(1) 実質一択スクリーニング**」を最優先評価に新設。判定フローの先頭で複数案の客観差を見て勝者採用する設計に変更。デフォルト方向を反転させた。

#### 結果（ドッグフード）

新フローを設計者自身の PR で適用（self-demonstration）:

- ベースライン PR #288: **要確認 81.25%**（過剰確認）
- 本 PR（新フロー適用）ドッグフード: **真の要確認 0/15 件**

7 件指摘中 6 件自動修正・1 件自動スキップ・要確認 0 件。**新フローが過剰確認を構造的に抑止できた最初の事例**。

#### 教訓

- **AI の「念のため確認」バイアスは構造で抑える**: チェック追加では効かない、デフォルト方向の反転で解く
- 「**else デフォルト方向が AI バイアスと同方向だと構造的欠陥が残る**」
- **複数項目の確認は俯瞰可能なファイル方式が優れる**: 連続 AskUserQuestion は user の認知負荷が高い
- **チームリーダーが過剰確認の自己再演を起こす**: 新フロー設計者本人が新フローを通すドッグフードで過剰確認を再演した

#### 数字・印象的フレーズ

- **要確認 81.25% → 0/15 件**
- 「else デフォルト方向が AI バイアスと同方向だと構造的欠陥が残る」
- 「実質一択スクリーニング」（最優先評価の新設）

#### Bluesky から見える作業との因果

**疲労の言語化（2026-02）**: 「確認フロー疲れ」という概念化の前段に、体感としての疲労があった。

> 「コード書いてないのになんでこんなに疲れるんだ 😇」
>
> 「仕事してないのに、働いてた時より疲労がやばい 😇 **AI を使って楽になるつもりが、AI に使われて苦しんでる気がする 😭**」

→ この疲労が「確認の総量自体が問題」という認識の起点になり、`/triage` 統一の動機を駆動。

**信頼破綻の決定打（2026-05）**:

### F10. データ契約 SSoT 2 軸 + 4 名チーム討議

- **期間**: 2026-04-17 〜 2026-05-04
- **LT 適性**: ★★★（技術深さあり、4 名討議の絵もユニーク）

#### 状況

F1（仕様書改革）で「**値の SSoT 階層**」（軸 1: pydantic Field > enums.yml > ConstrainedClient > 仕様書）は確立した。しかしそれだけでは不足が見えてきた。

- 同じ dto（`IngestResult` 等）の定義が複数ファイルに重複
- Port（Protocol）の所在がリポ間でバラバラ
- 公開 API 契約（HTTP / MCP / CLI）と内部表現の境界が曖昧
- `dict[str, Any]` が境界を越えて Core に侵入

**値の階層は解決しても、「契約がどこに 1 つだけ書かれているか」が解けていない**。これを軸 2 として整理する必要があった。

#### 打ち手 — AI 4 名チームで 40 項目 survey

過剰設計を避けつつ実効性のある SSoT 規約を作りたかったので、**異なる視点を持つ実在著者ペルソナ 4 名**でチームを編成:

- **Uncle Bob**（Clean Architecture の権化、原則駆動）
- **Martin Fowler**（リファクタリング・パターン、pragmatic）
- **Linus Torvalds**（実用主義・実装感覚）
- **Claude Shannon**（情報理論・SSoT の数学的定式化）

40 項目の設計トピックを survey させ、3 者独立収束した項目を採用候補とした。

**結果（独立収束した項目）:**

- 「SSoT 階層徹底」が最優先（軸 1 の強化）
- 「Port / dto / 公開契約の所在を 1 箇所に固定」（軸 2 の新設）
- **Clean Architecture の 4 層構造採用は却下**（過剰設計）
- Ports & Adapters の語彙のみ借用（用語の共通理解のため）

特筆: **Clean Arch の主唱者である Uncle Bob ペルソナ自身が「このプロジェクト規模では 4 層は過剰」と却下した**。普段なら Clean Arch を擁護する立場が拒否する流れは、AI 多角討議の威力を示す出来事だった。

#### 整備した SSoT 規約 — 2 軸併置

**軸 1: 値の競合解決階層**（既存）

| 優先度 | 情報源 | 役割 |
|---|---|---|
| 1 | pydantic Field 定義 | 型・制約値・デフォルト値 SSoT |
| 2 | `_schema/enums.yml` | Enum 定義 SSoT |
| 3 | ConstrainedClient | 解除不能ハードリミット |
| 4 | 仕様書 | 設計意図・Why |

**軸 2: 契約の所在（唯一性軸）**（新設）

| 契約種類 | 定義場所（推奨） | SSoT 表現 |
|---|---|---|
| 内部 contract（dto） | `<パッケージ>/models.py` | dataclass / TypedDict / pydantic BaseModel |
| Port（interface 契約） | `<パッケージ>/protocols.py` または `base.py` | Protocol / ABC |
| 公開 API contract | 各境界モジュール直下 | pydantic BaseModel / Discriminated Union |

これらは独立に機能する: 軸 1 は「値の真実」、軸 2 は「定義場所の唯一性」。問われる観点が別次元なので競合しない。

#### 仕様書群への展開

観点別に 4 ファイルへ分散配置（一括追加ではなく観点別、というユーザー直感に従う）:

- `spec-driven.md`: SSoT 2 軸併置の原則
- `architecture-guide.md`: Port / Adapter / dto / 5 境界カテゴリの語彙
- `coding-standards.md`: 型選択指針（dict[str, Any] 越境禁止等）
- `style-guide.md` §6.13: データ契約の記述ルール

#### 教訓

- 「**AI に独立人格を与えた多角討議で過剰設計を AI 自身が拒否する**」
- **SSoT は 1 軸では足りない**: 値（軸 1）と所在（軸 2）の 2 軸併置
- **語彙のみ借用は有効**: Ports & Adapters の 3 語だけ借りて全体構造は採用しない
- **観点別ファイル分散が読みやすい**: 一括追加すると過剰感が出る

### F11. Issue drift 検出ルーチン化

- **期間**: 2026-04-29 〜 2026-05-06
- **LT 適性**: ★★（具体的だが、地味）

#### 状況

rag-knowledge の epic #698（テスト整備）を進めようとして、Issue 本文を読み直したら違和感を覚えた。

「bluesky テストの mock 275 箇所」を「規模比例」（テスト数が多いから mock も多い）と解釈していたが、再評価すると違う:

> mock が 275 箇所もあるのは、**構造問題のサイン**ではないか？

調査すると確かに構造問題: テストが実装の内部状態を細かく覗いていて、構造が変わると一斉に壊れる。Issue は「mock を減らす」ではなく「**構造を変える**」だった。

AI 駆動開発で**自分が起票した Issue が、コード進化に追い付けず陳腐化**していた。これは AI 駆動の速度ゆえに常態化する課題。

#### 打ち手 — 着手前の前提検証ルーチン化

Issue 着手前に「Issue 本文と実コードの drift」を検出するルーチン化を方針として確立（`/plan-work` 前段への組み込みを提案）。

**4 カテゴリで drift を分類:**

| カテゴリ | 内容 |
|---|---|
| カテゴリエラー | Issue の問題定義そのものが間違っている |
| 陳腐化 | コードが進化して Issue が古くなった |
| 重複 | 他 Issue と内容が重なっている |
| 症状対処偏重 | 根本原因ではなく症状を対処しようとしている |

検出された場合は close → 再起票するか、Issue 本文を更新する。

epic #698 の例: 着手準備で drift を検出 → 関連 Issue 群を一旦 close → 新 epic として #700 → #701 → #702 を経て再定義し、最終的に **#725-#730 で再起票**。

#### 教訓

- **自分が起票した Issue が数ヶ月で陳腐化する**: AI 駆動開発で常態化する課題
- **着手前の前提検証ルーチン化**: AI に Issue を「そのまま実装」させない、必ず drift チェック
- **mock の規模を「構造問題のサイン」と読む**: 数値の比例関係に騙されない

#### 数字・印象的フレーズ

### F12. ルール疲れ現象 — advisory 遵守率 27% を計測

- **期間**: 2026-04-25 〜 2026-05
- **LT 適性**: ★★★★★（数値根拠付き失敗譚は強烈）

#### 状況

**2 日間（2026-04-25 〜 2026-04-26）で 9 件**の **メタ作業 Issue**（ルールについてのルールを整備する Issue）が agent-commons に集中していた。

- ルール A を守らせるために条文を追加
- でも守られないので条文 B を追加
- ルール間の矛盾を解消するため整合性ルール C を追加
- C が複雑すぎるので簡略化 Issue D
- ...

「既存ルールを守らせるためにさらにルールを追加する」という**ルール追加の連鎖**に陥っていた。ユーザー自身が「**同じような Issue が何度も出てきて改善されてない**」「ルール A よりも #262 の方が深刻ではないか」と直感的に気付いて、メタ的にルール疲れを問題化した。

個人セッションだけならまだしも、リーダー / メンバー / どの階層でも遵守率は変わらないのではないか？という疑問が湧いた。

#### 打ち手 — 遵守率の理論導出

Issue #274 のチーム討議で、advisory ルールの遵守率を **4 視点**から独立に検討:

| 視点 | アプローチ |
|---|---|
| 質的 | 過去セッションのジャーナルを定性レビュー |
| 量的 | ルール違反コミットの数 / 総コミット数 |
| 実証 | 「敢えてルール違反する PR」を投入して検出されるか測定 |
| 第 1 原理還元 | LLM の確率的生成特性から理論値を導出 |

**4 視点が独立に同じ結論に収束し、27% という理論値が出た**。

特に第 1 原理還元では:

```
0.9（ルール認識）× 0.5（インセンティブ）× 0.6（検証圧）= 0.27
```

3 係数いずれも advisory の世界での確率（DR 03 と呼ばれる DeepResearch 報告書由来）。**「ルール文書（advisory）で AI を制御するには 27% が理論上限」** という構造的根拠が確定した。階層非依存（リーダー / メンバー / 個人セッション問わず）。

#### 派生 — ドキュメント予防 → 設計予防

ルール疲れ現象は article-writer でも別形で再現していた:

- AI 生成記事の「balloon 内 backtick 誤用」をドキュメント警告で防ごうとして失敗
- 「キャラの一人称混入」をルール強化で防ごうとして失敗
- 何度もドキュメントを書き直しても、AI は別パターンで誤用を生成し続ける

最終的に**書き手（AI）の認知負荷を消す構造変更**で根治:

- `` `name` `` → 変換器が自動で `<code>` 化（balloon 内記法を AI が意識しなくていい）
- `:::l/:::r`（左右マーカー）→ `:::kuro-chan/:::nee-san`（キャラ名直結マーカー、AI が混乱しない）

「**ドキュメント警告は AI 予防として弱い**」という原則が複数面で再現された。

#### 教訓

- **advisory 遵守率は 27%**（数値で確定）
- 「**ルールを書くこと自体が問題の再生産**」— ルール追加で解こうとすると問題が増えていく
- 「**ドキュメント警告は AI 予防として弱い**」— 構造変更で書き手の認知負荷を消す
- **Hook / Skill / 構造的不可能化への移行**: ルールではなく仕組みで解く

#### 数字・印象的フレーズ

- **advisory 遵守率 27%**（4 視点で一致、階層非依存）
- 「**ルールを書くこと自体が問題の再生産**」
- 「**ドキュメント警告は AI 予防として弱い**」
- 「ドキュメント予防 → 設計予防」

#### Bluesky から見える作業との因果

**疲労が理論化の前にあった（2026-02）**: ルール疲れ・advisory 27% の理論導出は後付け。動機の起点は体感としての疲労だった。

> 「とりあえず疲労が蓄積してるので、明日はちょっと休めよう。**コード書いてないのになんでこんなに疲れるんだ 😇**」

→ 「コード書いてないのに疲れる」が問題提起となり、原因を 2 ヶ月かけて構造化した結果が **「advisory 遵守率 27%（理論導出値）」** という形に結実。

**自分で蒔いた種への気付き（2026-03）**: F2 と同じ戦略が裏目に出ていることを当事者として認識。

> 「遵守方針はまずは厳しめに書いて、運用で緩める条件を追加するやり方が良いかも」

→ この戦略を採用した結果、**ルール肥大化** → メタ作業 Issue 連鎖（2 日で 9 件） → Issue #301 で大規模棚卸し（290 行削減）に到達。**自分の方針判断が問題を生んでいた**ことを認めて方針転換する自己批判のサイクル。

> 「次から次へと環境改善案が出てきて**全然本題が進まないぞ、、 😇**」
>

→ 「**ハーネス改善作業自体が本題を侵食する**」という認識。Issue #301 ルール棚卸しで「常時ロードの削減」「禁止形のルール撤回」を進めた直接的な動機。

→ 副産物として「議論メンバー 1 人だけ召喚」方式が生まれる（チーム機構を使わない方が context 効率が良いという発見）

**2. SendMessage が届かない / リーダーしか動かない（2026-02）**

> 「チーム、毎回の性能ブレが大きいなぁ。議論はめちゃくちゃ活発で良い感じに機能する。で、そのまま実装させると、**リーダーしか動かない**とかになる。**議論と開発は別チームで進めないとダメ**そう」

→ SendMessage の `recipient` 指定で本文が消える OSS バグ発見、anthropics/claude-code Issue #25135 を起票
→ 暫定対応「`recipient: "team-lead"` 固定」運用

**3. リーダーがメンバー作業を把握していない（手戻り、2026-03）**

> 「リーダーはメンバーに任せて実装させてるから、リーダー自身が対応詳細を把握できてない。**解散後に指摘対応させると精度悪い。対応完全完了までチーム解散させちゃダメだな**」

→ 「チーム解散タイミングを `auto-finalize` → `merged` に移動」する仕様変更

**4. リーダー停止・ラリー詰まり（2026-04）**

- ルール棚卸し議論で kongming（リーダー）が「指示送付して idle」状態で停止 → team-lead が直接 broadcast
- QA 戦略実装で「**hermione が承認待ちで完全停止、shutdown_request 再送も応答なし。TeamDelete 不能で詰む可能性**」
- Issue #598 では方針往復が 3 回目に達し、メンバーから「方針を一度決めたら変えない約束をしてほしい」と抗議 → チーム解散・再整理

**5. 引き継ぎコスト・成果物消失**

- 親エージェントに改修内容・コンテキストが引き継がれず、解散後「なぜこの実装になったのか」を答えられない構造問題

#### 100M コンテキスト時代で復活、しかし...

> 「チーム組んでも全然コンテキストが尽きない」（2026-03、F4 と同期）

→ context 不足は解消されたが、**構造的問題（リーダーの把握不足・解散後の精度低下）は容量では解決しなかった**

#### 結論（現在の運用）

> 「**単一 AI の方がやってること把握できるから、こっちのほうがやっぱり良いかもなぁ。チームだと並行稼働が多すぎて何やってるか全貌をよく掴めない。あと、タスクがうまく回せなくて結局一人ずつしか動けてないとかもよくあるし**」
>
> 決定打: 「**結局最後の方はチーム使わなくなっちゃった。設計も、別窓で claude 立ち上げて、設計書.md 渡してみてもらったほうがコンテキストも分離できるし、客観性も上がるし、そっちが効率よい感じかも**」

**使い分け**:

- **チーム継続**: 並列実装（仕様確定後の独立タスク分割）
- **単独 AI に切り替え**: 議論・設計・レビューは別窓 Claude に単独投入

#### 教訓

- **チーム機構は context を増やしても根本問題は解決しない**: リーダーの把握不足・解散後の精度低下は構造的
- **「議論」と「実装」を同じチームで回すのは難しい**: 議論は活発でも、実装はリーダー集中
- **単独 AI + 別窓並行運用**の方が結局効率良い場面が多い
- **OSS のベータ機能は実用度の波が大きい**: 本番運用には早い段階あり

#### 数字・印象的フレーズ

- 「**ほんとチームでの開発は体験が良すぎるな**」（期待）
- 「**議論はめちゃくちゃ活発、でもそのまま実装させるとリーダーしか動かない**」（限界）

## 5. Claude の癖・苦手部分（カテゴリ別）

LT 聴衆（AI 開発実践者）に最も刺さるテーマ。

### 5.1 過剰確認バイアス系（最頻出）

- ユーザーが「進めて」「やって」と明確指示済みでも「進めますか？」型確認を発行
- `else デフォルト=要確認` の構造が「念のため確認」を温存
- **対策**: triage の「実質一択スクリーニング」を最優先評価に新設、抑止条件 S1〜S4
- **効かなかった**: チェック項目を増やす対策（v1）は「過剰確認に対しチェック追加は矛盾」と指摘され撤回

### 5.2 context bias 蓄積系（LT 核 ★4.4）

- 説教調・短い balloon 連発・過剰な感嘆詞
- AI 自身は自覚不能
- 対策: エージェント委譲リセット

### 5.3 不自然な会話・暗黙前提引っ張り系

- 「私のことは幸田さんで」型の不自然な自己紹介
- 配属初日のキャラに「過去仕事の資料を読んだ前提」で技術分析セリフを連発
- 社会的違和感は **AI 単独では検出不能の盲点**
- 対策: シーン設定で「人物の知識状態（初耳・既知）」を明示

### 5.4 タイトル提案・命名センス不足系

- 4-5 バリエーション出しても similar 構造の量産
- 抽象しすぎた言い換えで逆に分かりづらくなる（「印を 1 個だけ置く」< 「ロックファイル」）
- LLM にタイトル/キャッチコピーを要求すると**数値や誇張を盛る癖**
- 対策: 構造そのものを変える / user に叩き台を書いてもらう / タイトル生成を別レイヤに切り出す

### 5.5 advisory ルール無視・読まない系（LT 核 ★4.1）

- advisory 遵守率 **27%**（階層非依存）
- 対策: SessionStart Hook で自動注入

### 5.6 AskUserQuestion 品質・過剰発行系

- 「複数質問まとめて発行」廃止後も 4 質問一括発行で自己再演
- 1 質問に 2 軸混在
- 曖昧質問
- **効かなかった**: PreToolUse Hook で質問構造を機械検査する案は介入タイミングの問題で全撤回

### 5.7 ホワイトリスト/else デフォルトの構造的バイアス温存

- 真理値表＋擬似コードで形式的整合性を達成しても、**else デフォルト方向が AI バイアスと同方向だと構造的欠陥が残る**
- ペルソナ設定にセリフ具体例を書くと AI がそれに引きずられて画一化

### 5.8 scope drift / 指示外への暴走（LT 核 ★4.5）

- 「ついで」で勝手に直す
- turing メンバーが独断で実装、code-reviewer は「コード品質問題なし」で通してしまった（ユーザー指示整合性は観察外）

### 5.9 並列メンバーの独立性不足

- ペルソナ視点 + 「視点の明確な境界」がない場合は同じ結論への収束リスク
- 対策: 多忙ユーザー / 新規参加者 / 慎重派 / 構造批判 のような視点境界を明示

#### 検証予定の MCP 群（共通テーマ: AI が外部ツールを直接操作する）

| MCP | 何ができるか | 検証で見たいこと |
|---|---|---|
| **unityMCP** | Unity プロジェクトを MCP 経由で操作 | AI に Unity シーン編集・スクリプト生成を直接やらせる |
| **BlenderMCP** | Blender を MCP 経由で操作 | 3D アセット生成・編集を AI が直接行う |
| **Chrome DevTools MCP** | ブラウザを MCP 経由で操作（click / navigate / evaluate_script / screenshot / lighthouse 等） | Web アプリの自動テスト・パフォーマンス分析を AI が完結 |

#### 共通メッセージ

これら 3 つの MCP は表面的にはバラバラに見えるが、共通の方向性がある:

- **AI が外部ツールを「直接」操作する世界**: 従来「人間がツールを操作 → AI が支援」だったのが、「AI がツールを直接操作 → 人間がレビュー」に反転
- **MCP がインターフェースの統一を担う**: ツールごとに独自プロトコルではなく、MCP という共通言語で AI から触れる
- **ハーネスの守備範囲が拡大**: 今までは「コード書き」の制御だった Hook / 計画 / triage が、3D アセットや UI 操作にも適用される

#### LT で 30 秒触れるなら

```
→ ルール vs 仕組み の議論が、コードを超えて広がっていく
```
