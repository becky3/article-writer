## はじめに

Claude Code を使っていると、「あれ、いつこの設定入れたっけ？」「こんな仕様追加したっけ？」と過去のやり取りを後から確認したくなる場面があります。

実は Claude Code は、会話・ツール実行・ファイル編集のすべてを JSONL 形式でローカルに保存しているため、それを直接読めば追跡できます。

この記事では、保存場所、対象セッションの絞り込み、JSONL の構造、調査結果の時系列化、ハマりどころの 5 点を紹介します。本記事は **Windows 環境**で動作確認したものです。

## ログファイルの場所

Claude Code のセッションログは、ホームディレクトリ配下の以下のパスに JSONL（1 行 1 イベントの JSON）として保存されています。

```text
~/.claude/projects/<エンコード済みプロジェクトパス>/<セッションID>.jsonl
```

```bash
ls ~/.claude/projects/
```

ディレクトリ名・ファイル名から内容を直接読み取ることはできないため、次節の「対象セッションの見つけ方」で mtime とキーワードマッチを使って絞り込みます。

:::message
**補足: 命名規則と保存先のカスタマイズ**

命名規則:

- **ディレクトリ名**: プロジェクト（リポジトリ）の絶対パスから `:` `/` `\` を `-` に置換した形式です。たとえばプロジェクトが `C:/projects/sample-repo` にある場合は `C--projects-sample-repo` というディレクトリ名になります
- **ファイル名**: セッション ID（UUID）+ `.jsonl` です（例: `a1b2c3d4-e5f6-7890-abcd-ef0123456789.jsonl`）
- ディレクトリ名のエンコード規則は手動で組み立てるより、`ls` で列挙して目視特定するほうが安全です（仕様変更にも追従できます）

保存場所と書き込み有無を変える環境変数:

- `~/.claude` の場所は環境変数 `CLAUDE_CONFIG_DIR` を設定すると変更できます。設定している場合は配下の `projects/` も連動してそちらに移ります。Windows では `~/.claude` は `%USERPROFILE%\.claude` に解決されます。
- `CLAUDE_CODE_SKIP_PROMPT_HISTORY=1` を設定するとセッションログ自体が書き込まれなくなるため、このフラグを有効にしている場合は調査対象のセッションログが残っていない点に注意してください。

:::

## 対象セッションの見つけ方

プロジェクト配下には大量の JSONL が並んでいるので、まずは目的のセッションを 1 ファイルに絞り込みます。「時間範囲」と「キーワードマッチ数」の 2 段で絞ると速いです。

### 時間範囲を特定する

調査対象（PR 番号・Issue 番号・ファイルの特定行など）に応じて、関連する時刻を取得します。

```bash
# PR の作成時刻・マージ時刻
gh pr view <PR番号> --json createdAt,mergedAt,updatedAt

# Issue の場合は gh issue view <Issue番号> --json createdAt,closedAt

# 特定コミットの時刻
git log --format='%cI' <hash>

# 特定行を追加したコミットを特定
git blame -L <line>,<line> -- <file>
```

時刻ヒントが取れない場合は全 JSONL を候補にして、後段のキーワード絞り込みに頼ります。

### mtime で候補を絞り込む

ディレクトリを列挙し、対象時刻をカバーする mtime のファイルだけ残します。

```bash
ls -la ~/.claude/projects/<エンコード済みプロジェクトパス>/*.jsonl
```

### キーワードマッチ数で本命を特定する

調査対象のキーワード（Issue 番号、機能名、対象文言など）で固定文字列マッチ件数をランキングし、突出したファイルを本命として特定します。`grep -F` は正規表現解釈を抑止するので、`#123` のような特殊記号を含むキーワードもそのまま渡せます。

```bash
for f in ~/.claude/projects/<エンコード済みプロジェクトパス>/*.jsonl; do
  count=$(grep -F -c "<keyword>" "$f")
  echo "$count  $f"
done | sort -rn | head -10
```

実例として、[後述する事案](#実例-ログを読んだら何が分かったか)で「Issue 本文に存在しない制約が独自追加された経緯」を調査したとき、候補 6 件中で本命のみマッチ 210 件、他は 0〜4 件と分布が極端に偏ったため、迷わず 1 ファイルに絞り込めました。

## JSONL の構造を読み解く

調査時に押さえるべき主要フィールドは以下です。

| フィールド | 役割 | 値の例 |
|---|---|---|
| `type` | イベント種別。発話者がユーザーか AI かの判定に使う | `user` / `assistant` |
| `message.role` | `type` と同じく発話者を示す。冗長だが両方記録される | `user` / `assistant` |
| `message.content` | 発話本文。配列（複数 part）または **文字列** で入る | 文字列／配列 |
| `message.content[].type` | 配列要素の中身が何か（生テキストかツール呼び出しか結果か） | `text` / `tool_use` / `tool_result` |
| `timestamp` | イベント発生時刻。時系列に並べる際のキー | ISO 8601（UTC） |

ログには複数の種類のイベントが混ざって記録されています。「使うもの」「除外するもの」「取り方に注意が必要なもの」に分けて整理します。

### 使うイベント

| イベント | 判別 | 用途 |
|---|---|---|
| ユーザー発話 | `type: user` | 会話の流れ・承認の有無の確認 |
| AI 発話 | `type: assistant` かつ `content[].type: text` | 会話の流れ・AI の説明や提案の確認 |
| ツール呼び出し | `content[].type: tool_use` | 対象文言を追加した編集の特定 |
| ツール結果 | `content[].type: tool_result` | 編集の成否や対象範囲の確認 |
| システム介入 | `message.content` の文字列値が `[Request interrupted by user]` などのシステム文字列 | AI の発話が途中で打ち切られた状況の判定 |

### 除外するイベント

`content[].text` が `Base directory for this skill: ...` で始まるイベントは、スキル起動時にシステムが注入したテキストです。ユーザー・AI どちらの発話でもないため、会話の流れに混ぜずに除外します。

### 取り方の注意点

- `message.content` は配列と文字列の両方で入ります。`"ok"` のような短い応答は文字列で来るので、両方の型に対応してパースします
- ツール記録は `tool_use`（呼び出し）と `tool_result`（結果）の 2 イベントに分かれます。AI が対象文言を追加した箇所を探すときは `tool_use` 側を見ます

## Windows 環境の注意点

ここまでの手順を Windows + Git Bash で実行したときに踏みやすい落とし穴を補足します。

### Git Bash の `cat` / `grep` は文字化けする

Git Bash 上で JSONL の本文を読むと、日本語が文字化けして対象文言が grep にヒットしないことがあります。**JSONL 本文の読解は PowerShell で UTF-8 を明示**して読み込むのが安定します。

```powershell:read-jsonl.ps1
# $path は対象 JSONL のフルパス。下のスニペットは読み込み部分のパターン例
$lines = [System.IO.File]::ReadAllLines($path, [System.Text.Encoding]::UTF8)
foreach ($line in $lines) {
  if (-not $line) { continue }
  $obj = $line | ConvertFrom-Json
  # ... 処理 ...
}
```

`ConvertFrom-Json` で 1 行ずつパースし、前述の `content` 型チェック（配列か文字列か）を必ず挟みます。

## 対象箇所を抽出して時系列化する

ここから先で出てくる「対象文言」は、調査で追跡したい文字列を指します。

本命セッションを 1 ファイルに絞れたら、対象文言を追加した編集の前後を時系列に並べます。これにより、その編集が会話のどの流れで行われたかが見えるようになります。

具体的には、対象文言で grep してその文言を追加した `tool_use` を特定し、**その前後 50〜100 行**のユーザー発話・AI 発話・ツール実行を抜き出します。範囲はセッションの密度で調整してください。列構成は以下を推奨します。

| 列 | 内容 |
|---|---|
| 行番号 | JSONL 内の行番号 |
| 時刻 | `timestamp` を JST に換算 |
| 種別 | USER / AI / TOOL / SYSTEM |
| 要約 | 内容の短い要約 |

このテーブル上で「ユーザー発話 → AI が質問 → ユーザー応答 → AI のツール実行」の流れがつながるかを確認します。AI が質問を投げた直後にユーザー応答なしで次のツール実行に進んでいる箇所があれば、独断進行のサインです。

## 実例: ログを読んだら何が分かったか

ログを読んで構造的な対策まで繋げられた事例として、筆者が遭遇したケースを紹介します。

### 起きたこと

ある PR で、Issue に書かれていない制約「投稿内 URL の自動取り込みは実行しない」が仕様書と実装に紛れ込んでいました。

その PR の本来の目的は、取り込みに失敗した SNS 投稿を URL 指定で 1 件だけ取り込み直すピンポイント修復機能の追加でした。修復用途なので投稿本文のリンク先まで含めて取り込み直したいのに、AI が独断で「リンク先は取り込まない」と範囲を狭めていたのです。

しかも制約には「リンク先の取り込みは別の責務」というもっともらしい理由付けまで添えられていたため、AI レビューも素通りし、ユーザー確認もなくマージされていました。

### ログを読んで分かったこと

「なぜこの一文が入ったのか」は、PR 本文・Issue・コミットメッセージ・レビューコメントのどこを読んでも分かりませんでした。残された手がかりは、そのときのセッションログだけ。JSONL を解析した結果、次のことが判明しました。

- 制約はユーザーとの会話に一度も登場せず、AI が仕様書執筆時に独断追加していた
- 制約に添えられた理由付けは AI が単独で生成したもので、ピンポイント修復という機能本来の目的と矛盾していた

コミット履歴や PR コメントには現れない種類の情報です。

### 調査結果が次に繋がった対応

経緯が特定できたことで、対症療法ではなく構造的な対策に踏み込めました。具体的には次の 3 つを開発ルール・レビュー観点に取り込みました。

- Issue にない制約を AI が追加するときは、必ずユーザー確認を取る
- AI が「なぜそう書いたか」の理由付けを書く前に、Issue の目的と整合しているか AI がチェックする
- レビューでは「理由付けが書かれている＝良い」で止めず、その理由付けの中身まで踏み込んで検証する

---

ログを読まずに「制約を消して終わり」にしていたら、同じパターンが別 PR で再発していたはずです。「なぜそうなったか」の一次資料が手元にあるという状態が、対策のレベルを変えてくれます。

## まとめ

- Claude Code のセッションログは `~/.claude/projects/<エンコード済みプロジェクトパス>/*.jsonl` に保存されている
- 対象セッションの絞り込みは **時間範囲特定 → mtime 絞り込み → キーワードマッチ数ランキング** の 3 段、その後 **対象の編集箇所を抽出 → 時系列テーブル化** で読み解く
- `content` の型（配列／文字列）チェック、スキル自動ロードの除外、システム介入の取り扱いに注意する
- Windows では PowerShell + UTF-8 の明示読み込みが安定

セッションログは AI の意思決定プロセスを後追いできる一次資料です。「テストもレビューも通ったのになぜ？」という事象の真因にたどり着く手段として活用できます。
